# Deploying without coding using the deepagents-cli
 
 The deepagents-cli allows you to deploy a powerful agent with no coding. It does this by being opinionated about the feature set and the agent project's structure. Concretely, instructions for the Agent are provided by an `AGENTS.md` and skills files located in specific directories in the project structure. Tools are provided by MCP servers defined in an `mcp.json`. Subagents are defined in subfolders in the project structure. A deepagents.toml file further defines agent features, the agent name, models, auth, frontend, and sandboxes. With this information defined, the deepagents-cli bundler builds a deployment. 

The bundler builds an agent using `create_deep_agent(...)` with a `CompositeBackend`, `MemoryMiddleware`, `SkillsMiddleware`, and an optional sandbox. With that middleware, the agent knows how to use a file system, find instructions in memory, and skills in skills files. You can't insert custom middleware, swap the backend, or restructure the flow. The agent includes a:

- **`/memories/` mount point.** The template hardcodes `MEMORIES_PREFIX = "/memories/"`. All paths the agent sees (`/memories/AGENTS.md`, `/memories/skills/...`, `/memories/user/...`) are imposed by this constant.
- **Permission defaults.** The CLI applies fixed permissions: `/memories/AGENTS.md` and `/memories/skills/**` are read-only; `/memories/user/**` is per-user writable; subagents are isolated. 

You may notice this structure is similar to the deep_tutor structure. This was intentional. `deep_tutor` has a small amount of custom code to load information based on the assistant and namespace, but otherwise relies on the same set of middleware.

**When to use the CLI path vs the code path:** Use the CLI when your agent fits the standard `create_deep_agent` shape — system prompt, skills, optional MCP tools, and subagents. Use the code path (writing `agent.py` and `langgraph.json` yourself, as in `deep_tutor`) when you need custom middleware, a different backend, or a graph structure that doesn't fit this shape.

In the remainder of this lesson, you'll learn more about how to structure a project to deploy a powerful and flexible no-code agent.

## Project Structure

A deepagents project is a directory with a convention-based layout. The CLI discovers files by name — no registration required.

```
my-agent/
├── deepagents.toml    # Agent config (required)
├── AGENTS.md          # System prompt (required)
├── .env               # API keys
├── mcp.json           # MCP servers (optional)
├── skills/            # Skills directory (optional)
├── subagents/         # Sub-agents (optional)
└── user/              # Per-user writable memory (optional)
```

Two files are required. Everything else is optional and loaded only if present.

---

## deepagents.toml

The main config file. It has five sections:

```toml
[agent]
name = "my-agent"
model = "anthropic:claude-sonnet-4-6"

[auth]
provider = "anonymous"

[frontend]
enabled = true
app_name = "My Agent"

[sandbox]
provider = "langsmith"
scope = "thread"

[memories]
backend = "store"
```

| Section | Required | What it configures |
|---|---|---|
| `[agent]` | Yes | Deployment name and model |
| `[auth]` | No | Who can call the agent (`anonymous`, `supabase`, `clerk`) |
| `[frontend]` | No | Deploy a built-in React UI |
| `[sandbox]` | No | Code execution environment for the agent |
| `[memories]` | No | Backing store for `/memories/` (`AGENTS.md`, `skills/`, `user/`) |

**`[agent]`** is the only required section. Name and model use the same `provider:model` format as `init_chat_model` in LangGraph.

**`[auth]`** controls access. `anonymous` deploys an open endpoint — useful for demos and internal tools. `supabase` and `clerk` gate the API with real user tokens; threads and memory are automatically scoped per user. Without an `[auth]` section, the deployment falls back to LangSmith's default `x-api-key` requirement (pass your LangSmith API key in that header). With Supabase or Clerk, clients pass the user's session token in the `Authorization: Bearer <token>` header.

**`[frontend]`** deploys a React chat UI alongside the agent API. Set `app_name` to customize the header.

**`[sandbox]`** provisions a code execution environment for the agent. The `langsmith` provider uses LangSmith's managed sandbox; `daytona`, `modal`, and `runloop` are third-party options. `scope = "thread"` creates a fresh sandbox per conversation; `scope = "assistant"` keeps one sandbox alive across all conversations.

**`[memories]`** chooses where `/memories/` content (`AGENTS.md`, `skills/`, `user/`) is stored. `backend = "store"` uses the deployment's LangGraph Store (the same Postgres that backs threads). `backend = "hub"` (the default) uses a remote LangSmith Hub repo with persistent, git-like versioning across deployments.

## AGENTS.md

The system prompt. It is loaded at the start of every session and stays in context for the entire conversation. Write it the same way you would write a system prompt for any agent — instructions, persona, constraints.

```markdown
# Agent Instructions

You are a helpful assistant. Answer questions clearly and concisely.
Keep your responses focused on what the user asked.
```

AGENTS.md is read-only at runtime in the main agent. Skills can extend the agent's knowledge without changing the base prompt.

---

## mcp.json

Configures MCP (Model Context Protocol) servers — external tools the agent can call.

```json
{
  "mcpServers": {
    "docs": {
      "url": "https://docs.langchain.com/mcp",
      "transport": "http"
    }
  }
}
```

One constraint: **only HTTP and SSE transports are supported**. Stdio transports (which spawn a local process) are rejected at bundle time — there is no local process to spawn in a deployed container. Every MCP server must be reachable over HTTP.

---

## skills/

Each subdirectory under `skills/` is a skill, and its `SKILL.md` file contains instructions the agent loads on demand.

```
skills/
└── module-1/
    ├── SKILL.md        # Teaching instructions for this module
    └── information.md  # Reference material
```

The agent reads skill files at runtime when needed. Skills let you update agent knowledge without touching the system prompt.

---

## subagents/

The `subagents/` directory holds specialized child agents that the main agent can delegate to. Each subdirectory is a subagent with its own configuration:

```
subagents/
└── researcher/
    ├── deepagents.toml    # Subagent's own config (name, model)
    ├── AGENTS.md          # Subagent's system prompt
    └── skills/            # Optional skills scoped to this subagent
```

At bundle time, the CLI auto-discovers each subagent directory. The main agent receives a `task` tool that dispatches work to a subagent by name. Subagents run in isolation — each has its own state, messages, and (optional) skills, separate from the main agent's.

Use subagents when you want to split responsibilities: a main agent that orchestrates, and specialized subagents that handle narrow tasks (research, analysis, etc.) with their own prompts and skills.

---

## user/

The `user/` directory is per-user writable memory. If `AGENTS.md` exists inside `user/`, it seeds as a template for each new user; otherwise the agent starts with an empty file. The agent can read and write files under `user/` at runtime — changes persist across sessions for that user.

Each user gets their own namespace, derived from the authenticated identity set by `[auth]`. Once a user's `user/AGENTS.md` is seeded on first access, it is **never overwritten on redeploy** — agent edits persist across deployments.

This is the mechanism for building agents that learn and adapt to individual users over time.

<details>
<summary><strong>User Auth Details</strong></summary>

Since you just learned about the details of auth in a prior module, you can appreciate the details of how auth is implemented here.

Once `[auth]` is configured, every request to the deployment goes through a two-stage auth flow:

1. **Request hits the deployment.**
2. **`@auth.authenticate` runs** (the provider-specific block generated from `[auth].provider`). It validates credentials and produces a `BaseUser` exposed as `ctx.user`, with `ctx.user.identity` set to the authenticated user ID. (This same `BaseUser` will also be exposed as `runtime.server_info.user` later, when the graph executes — see step 5 and store scoping.)
3. **Auth gate passes** → request proceeds to the actual handler (e.g., create a thread, list threads).
4. **Before the handler executes, `@auth.on.threads` runs** with the same `ctx`. The CLI generates this from a shared template that does two things:
   - **For writes** (create thread, save state): tags the resource with `metadata.owner = ctx.user.identity`.
   - **For reads** (list threads, get thread): returns `{"owner": ctx.user.identity}` as a filter — only resources matching that filter are visible.
5. **The actual handler runs against the filtered/tagged resource set.** If the graph then executes, `runtime.server_info.user` is also available inside nodes — same authenticated identity, just exposed via a different attribute inside graph execution.

Per-user thread isolation is therefore automatic in CLI deploys: User A's threads have `metadata.owner = "userA"`, User B's filter is `{"owner": "userB"}`, and B never sees A's threads.

**Store scoping works differently.** The CLI template only generates `@auth.on.threads` — there is no `@auth.on.store`. Per-user `/memories/user/**` isolation happens at *execution* time: when the agent calls `read_file("/memories/user/AGENTS.md")`, the `CompositeBackend` routes the call to `StoreBackend`, which reads `runtime.server_info.user.identity` to determine the namespace. The actual namespace tuple is `(assistant_id, user_identity)` — so user data is scoped *per-assistant per-user*. The same user using two different assistants in the same deployment gets two separate namespaces; data they wrote via one assistant is invisible to the other.

| Phase | What's scoped | Where the check lives |
|---|---|---|
| Auth phase (pre-graph) | Thread visibility | `@auth.on.threads` (CLI-generated), reads `ctx.user.identity` |
| Execution phase (inside graph) | Per-user Store namespace (`/memories/user/**`) | `StoreBackend` inside `deepagents`, reads `runtime.server_info.user.identity` |

LangSmith Studio bypass: `is_studio_user(ctx.user)` returns an empty filter so Studio sees everything for debugging.

</details>

## The Three Commands

The CLI is a project dependency in this course (in `python/pyproject.toml`), so install it by syncing the project venv:

```bash
cd /Users/geoffladwig/Documents/Github/lca-deployment/python
uv sync
```

Then invoke the CLI via `uv run`:

```bash
uv run deepagents init [name]              # Scaffold a new project
uv run deepagents dev [--port 2024]        # Local development server
uv run deepagents deploy                   # Build and deploy to production
```

Running via `uv run` ensures the CLI and its `langgraph` subprocess both use the project's pinned versions (`python/uv.lock`). For your own projects you can also install globally with `uv tool install deepagents-cli`, but it's then your responsibility to keep the global tool's `langgraph-cli` version aligned with whatever your project expects.

**`init`** creates a new project directory with the required files. Use it to start fresh.

**`dev`** starts a local server you can use to verify the agent works before deploying. The dev server exposes the same API as the production deployment.

**`deploy`** builds a new image and creates a production revision. Every `deploy` call is a fresh deployment — it does not update in place. Use `dev` for iteration; `deploy` when you're ready to ship.

## What the deployment exposes

Once deployed, the agent runs on LangSmith's Agent Server runtime and exposes 30+ endpoints, grouped roughly as:

- **Agent Protocol** — standard API for invoking the agent (threads, runs, state) and building UIs
- **MCP** — call your deployed agent as a tool from other agents
- **A2A** — multi-agent orchestration via the A2A protocol
- **Human-in-the-loop** — approval gates for sensitive actions
- **Memory** — short-term (Checkpointer) and long-term (Store) memory access

Every deployment also exposes `/docs` (Swagger UI) with the complete endpoint list. For the full API reference, see the [LangSmith Agent Server API](https://docs.langchain.com/langsmith/server-api-ref).

Clients invoke the deployment using the same `langgraph-sdk` patterns from module 2: create a client pointing at the deployment URL, create a thread, submit a run. The CLI's default assistant ID is `"agent"`. Pass the appropriate auth header (`x-api-key`, `Authorization: Bearer <token>`, or none) based on your `[auth]` configuration.
