# Records: Assistants, Threads, Runs

The deployment's database holds three things you've already met — the **checkpointer** (graph state), the **Store** (long-term memory), and a third category we've been pointing at without naming: the **records** that describe how runs are parameterized and what's happened.

The records are split into three tables: **assistants**, **threads**, and **runs**. This lesson walks through what's in each one and how they combine when a run executes.

---

## The Three Records

| Record | Identified by | Holds | Created by |
|---|---|---|---|
| **Assistant** | `assistant_id` | `graph_id`, `name`, `context`, `config`, `metadata`, `version` | `client.assistants.create(...)` — every deployed graph also auto-creates a default assistant. |
| **Thread** | `thread_id` | metadata, status, timestamps | `client.threads.create()` |
| **Run** | `run_id` | `thread_id`, `assistant_id`, status, `kwargs` (input + per-run config), metadata, `multitask_strategy`, timestamps | `client.runs.wait(...)` / `runs.stream(...)` |

The records live in the same Postgres database as the checkpointer and the Store, but they're separate concerns:

- **Checkpointer** stores *what the agent thinks* — graph state at each superstep.
- **Store** stores *what the agent remembers across threads* — long-term key-value memory.
- **Records** store *how the agent was asked to think and what's happened* — assistant configuration, thread metadata, run history.

---

## How a Run Uses the Records

A single call to `client.runs.wait(thread_id, assistant_id, input=...)` touches all three records:

```text
client.runs.wait(thread_id, assistant_id, input=...)
                       ↓
   1. Look up the assistant by assistant_id  → pulls its context and config
   2. Load thread state by thread_id          → resumes the conversation (via the checkpointer)
   3. Create a new run record                 → status="pending"
                       ↓
   Worker calls graph.invoke(input, config) with the assistant's context
                       ↓
   4. Append checkpoints to the thread        → output lands in thread state
   5. Update the run record                   → status="success"
```

Two IDs do the lookup work:
- `assistant_id` resolves the graph's parameterization — *how* the graph should behave for this call.
- `thread_id` resolves the conversation state — *which* conversation this run extends.

The run record itself is a log of *what was asked and how it finished* — input, status, timing. The output isn't stored on the run; it lives in the thread as the final checkpoint, and you read it back via `threads.get_state` or `runs.join`.

---

## What's on an Assistant: `context` vs `config`

The assistant record holds two distinct slots for parameterization. They're often conflated, so it's worth pinning down:

| Slot | Purpose | Schema | Typical contents |
|---|---|---|---|
| `context` | Typed runtime values the graph *logic* reads | Defined by the graph's `context_schema` | `module_id`, `store_namespace`, `user_id` |
| `config` | Execution plumbing for the LangGraph runtime | Free-form `RunnableConfig` | `recursion_limit`, `tags`, `metadata`, `configurable` |

Both are set at `assistants.create()` and merged with per-run overrides at execution time. The difference is which side reads them: `context` is for your code, `config` is for the framework.

## SDK Access

### Assistants

<Tabs>
  <Tab title="Python">
```python
# Create
assistant = await client.assistants.create(
    graph_id="tutor",
    name="Tutor — Module 1",
    context={"module_id": "module-1", "store_namespace": ""},
)

# Read
assistant = await client.assistants.get(assistant["assistant_id"])

# Update (replaces context, does not merge)
await client.assistants.update(assistant["assistant_id"], context={...})

# Search / Delete
results = await client.assistants.search(graph_id="tutor")
await client.assistants.delete(assistant["assistant_id"])
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// Create
const assistant = await client.assistants.create({
  graphId: "tutor",
  name: "Tutor — Module 1",
  context: { module_id: "module-1", store_namespace: "" },
});

// Read
await client.assistants.get(assistant.assistant_id);

// Update (replaces context, does not merge)
await client.assistants.update(assistant.assistant_id, { context: {...} });

// Search / Delete
await client.assistants.search({ graphId: "tutor" });
await client.assistants.delete(assistant.assistant_id);
```

</Tabs>

## Recap

- The deployment persists three record types — **assistants** (config + context), **threads** (conversation slots), and **runs** (execution history) — alongside the checkpointer and the Store, but as separate tables.
- A run is addressed by two IDs: `assistant_id` resolves the graph's parameterization (context, config), `thread_id` resolves the conversation state (via the checkpointer).
- An assistant holds two slots for parameterization: `context` for typed values your graph *logic* reads, and `config` for `RunnableConfig` plumbing (`recursion_limit`, `tags`, …).
