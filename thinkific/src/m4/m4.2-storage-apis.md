In this lesson you'll use the LangGraph SDK to access two of the three storage types from the previous lesson: read and write the **Store**, then read thread state from the **Checkpointer**. You'll use the `deep_tutor` deployment you already have running. (The third type — the container's local filesystem — isn't reachable from the SDK. It is updated by deploying a new image.)

Accessing storage from outside the deployment is useful for pre-loading Store content (seed data, configuration), building admin tools that read or edit data without invoking the agent, or inspecting the Checkpointer to debug or audit conversations. In `deep_tutor`, the UI's `/register` and `/login` routes use the Store to save and look up student profiles; the Checkpointer holds the conversation history that lets students resume threads across browser sessions.

By the end, you will have:
- Written a student profile to the Store and read it back
- Sent a message to the agent and read the resulting thread state from the Checkpointer

### Read/Write to Store

The Store is a key-value store. Each item is identified by a **namespace** (a tuple of strings) plus a **key** (a string). The namespace scopes data to a logical entity — for example, one namespace per user.

The SDK exposes three core operations:

- `put_item` / `putItem` — write or upsert an item
- `get_item` / `getItem` — read a single item
- `search_items` / `searchItems` — list every item under a namespace prefix

### Writing a profile

In `deep_tutor`, each student's data is stored under their own namespace. The UI builds the namespace from the student's email by replacing `.` with `_` (e.g. `jane@example.com` → `("jane@example_com",)`). The script below uses a hardcoded `("john_doe",)` namespace as a stand-in.

<Tabs>
  <Tab title="Python">
```python
# python/m4/l2/store.py
NAMESPACE = ("john_doe",)

await client.store.put_item(NAMESPACE, key="profile", value={
    "first_name": "John",
    "last_name": "Doe",
    "email": "john@example.com",
    "goals": "Understand how LangGraph deployments work end to end.",
})
print(f"Wrote profile to Store under namespace {NAMESPACE!r}")
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m4/l2/store.ts
const NAMESPACE = ["john_doe"];

await client.store.putItem(NAMESPACE, "profile", {
  first_name: "John",
  last_name: "Doe",
  email: "john@example.com",
  goals: "Understand how LangGraph deployments work end to end.",
});
console.log(`Wrote profile to Store under namespace ${JSON.stringify(NAMESPACE)}`);
```
  </Tab>
</Tabs>

```text
Wrote profile to Store under namespace ('john_doe',)
```

`put_item` / `putItem` is an upsert — writing the same namespace + key again overwrites the previous value.

### Reading thread state

After the run completes, `get_state` / `getState` returns the last checkpoint — the full state as it was after the final node, including the complete message history.

<Tabs>
  <Tab title="Python">
```python
# python/m4/l2/checkpointer.py
state = await client.threads.get_state(thread["thread_id"])
messages = state["values"].get("messages", [])
print(f"Checkpointer — thread has {len(messages)} messages:")
for msg in messages:
    role = msg.get("type", msg.get("role", "?"))
    content = msg.get("content", "")
    print(f"  [{role}]: {str(content)[:80]}")
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m4/l2/checkpointer.ts
const state = await client.threads.getState(thread.thread_id);
const messages = ((state.values as Record<string, unknown>)?.messages ?? []) as Array<Record<string, unknown>>;
console.log(`Checkpointer — thread has ${messages.length} messages:`);
for (const msg of messages) {
  const role = msg.type ?? msg.role ?? "?";
  let content = msg.content;
  if (Array.isArray(content)) {
    content = (content[0] as Record<string, unknown>)?.text ?? "";
  }
  console.log(`  [${role}]: ${String(content).slice(0, 80)}`);
}
```
  </Tab>
</Tabs>

```text
Checkpointer — thread has 6 messages:
  [human]: What is the checkpointer in LangGraph?
  [ai]: 
  [tool]:      1  ---
               2  name: module-1
               3  description: Teaching instructions for M
  [ai]: 
  [tool]:      1  # Module 1 — Reference Material
               2
               3  ## The Three Layers
  [ai]: A checkpointer in LangGraph is a persistence layer that saves graph state
```

### Reading thread history

`get_history` / `getHistory` returns every checkpoint saved during the run — one per superstep. This shows how state evolved step by step through the graph.

<Tabs>
  <Tab title="Python">
```python
# python/m4/l2/checkpointer.py
history = await client.threads.get_history(thread["thread_id"])
print(f"Checkpointer history — {len(history)} checkpoints saved for this thread")
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m4/l2/checkpointer.ts
const history = await client.threads.getHistory(thread.thread_id);
console.log(`Checkpointer history — ${history.length} checkpoints saved for this thread`);
```
  </Tab>
</Tabs>

```text
Checkpointer history — 10 checkpoints saved for this thread
```

A single message exchange through `deep_tutor` saves around 10 checkpoints — one per superstep, covering the agent's LLM step, each tool execution, and the final response.

## Recap

- The Store is a namespace + key keyed store. `put_item` / `get_item` / `search_items` cover the core read/write/list operations from outside the deployment.
- The Checkpointer is readable and writable from the SDK. `threads.get_state` returns the latest checkpoint; `threads.get_history` returns every checkpoint saved for the thread; `threads.update_state` writes a new checkpoint directly — the mechanism behind human-in-the-loop state edits. LangGraph also writes it automatically during runs.
- Both are accessible from outside the deployment via the SDK, gated by the deployment's auth rules.
