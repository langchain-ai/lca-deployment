# Built-in and Custom Routes

Every LangSmith deployment exposes a set of HTTP endpoints. Most are *built-in* — the Agent Server API that lets clients manage runs, threads, assistants, and the store. You can also extend the deployment with *custom* endpoints — your own routes mounted alongside the built-in ones. This is how the tutor UI is served directly from the deployment with no separate server.

---

## Built-in routes: the Agent Server API

Every deployment runs the LangSmith Agent Server runtime, which exposes 30+ endpoints grouped roughly as:

- `/runs/*` — start and manage runs
- `/threads/*` — manage conversation threads and checkpoints
- `/assistants/*` — create, list, update, delete assistants
- `/store/*` — read/write key-value memory
- `/mcp/*` — Model Context Protocol resources
- plus health, auth, and metadata endpoints

You have already used `/runs`, `/threads`, and `/assistants` indirectly via the LangGraph SDK in m2.1 and m2.2 — those SDK calls map to HTTP requests against these built-in routes.

### Browsing all routes

Every deployment also exposes a Swagger UI at `/docs` that lists every available endpoint with request/response schemas — built-in and custom together:

```text
{deployment-url}/docs
```

For local development:

```text
http://localhost:2024/docs
```

For the full schema, see the [Agent Server API reference](https://docs.langchain.com/langsmith/server-api-ref).

## Custom routes

The `http.app` field in `langgraph.json` points to a web application object. LangSmith mounts it alongside the built-in routes, so your endpoints live at the same URL as the agent API.

<!-- block: m2.3-langgraph-json -->
<Tabs>
  <Tab title="Python">
```json {6-7}
{
  "dependencies": ["."],
  "graphs": {
    "tutor": "./agent/agent.py:graph"
  },
  "http": {
    "app": "./ui/app.py:app"
  },
  "env": "../.env"
}
```
  </Tab>
  <Tab title="TypeScript">
```json {6-7}
{
  "dependencies": ["."],
  "graphs": {
    "tutor": "./agent/agent.ts:graph"
  },
  "http": {
    "app": "./ui/app.ts:app"
  },
  "env": "../.env"
}
```
  </Tab>
</Tabs>
<!-- /block: m2.3-langgraph-json -->

The docs describe this as "a Starlette app in Python or a Hono app in JavaScript". The tutor server uses FastAPI (Python) and Hono (TypeScript).

### The tutor's custom routes

The tutor server adds several endpoints on top of the built-in Agent Server API:

| Route | Method | Purpose |
|---|---|---|
| `/` | GET | Serves `index.html` |
| `/modules` | GET | Returns the module list |
| `/register` | POST | Creates student profile, assistants, and threads |
| `/login` | POST | Looks up an existing student by email |
| `/chat` | POST | Streams a response from the agent |
| `/resources` | GET | Returns all assistants and threads (for debugging) |

These appear in `/docs` alongside the built-in routes.

## The deployment URL challenge

The server uses the LangGraph SDK to forward requests to the agent — which means it needs to know the deployment URL. It can't be hardcoded: the URL is different locally, in staging, and in production.

One approach: derive the URL from the incoming HTTP request. In cloud deployments, TLS is terminated at the load balancer, so the raw request URL reflects only the internal network. The `X-Forwarded-Host` and `X-Forwarded-Proto` headers carry the original public-facing values.

This logic is isolated in `deployment.py` / `deployment.ts` so it's easy to replace if needed:

<!-- block: m2.3-deployment -->
<Tabs>
  <Tab title="Python">
```python
# python/deep_tutor/ui/deployment.py
from fastapi import Request

def get_deployment_url(request: Request) -> str:
    host = request.headers.get("x-forwarded-host") or request.headers.get("host")
    scheme = request.headers.get("x-forwarded-proto") or request.url.scheme
    return f"{scheme}://{host}"
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/deep_tutor/ui/deployment.ts
import type { Context } from "hono";

export function getDeploymentUrl(c: Context): string {
  const host = c.req.header("x-forwarded-host") || c.req.header("host") || "";
  const scheme =
    c.req.header("x-forwarded-proto") ||
    new URL(c.req.url).protocol.replace(":", "");
  return `${scheme}://${host}`;
}
```
  </Tab>
</Tabs>
<!-- /block: m2.3-deployment -->

Each route handler calls `get_deployment_url` / `getDeploymentUrl` to create a LangGraph SDK client pointed at the deployment:

<!-- block: m2.3-chat-route -->
<Tabs>
  <Tab title="Python">
```python
# python/deep_tutor/ui/app.py
@app.post("/chat")
async def chat(request: Request, req: ChatRequest):
    client = get_cached_client(get_deployment_url(request))
    return StreamingResponse(
        stream_response(client, req.thread_id, req.assistant_id, req.message),
        media_type="text/plain",
    )
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/deep_tutor/ui/app.ts
app.post("/chat", async (c) => {
  const { message, assistant_id, thread_id } = await c.req.json();
  const client = getClient(getDeploymentUrl(c));

  return stream(c, async (s) => {
    for await (const chunk of streamResponse(client, thread_id, assistant_id, message)) {
      await s.write(chunk);
    }
  });
});
```

## Recap

- Every deployment runs the Agent Server API — built-in HTTP endpoints for runs, threads, assistants, store, and more. Visit `{deployment-url}/docs` to browse the full list.
- `http.app` in `langgraph.json` registers a web app as **custom** routes alongside the built-in ones.
- The tutor server adds `/`, `/modules`, `/register`, `/login`, `/chat`, and `/resources`.
- Custom routes that call back into the same deployment need to discover the deployment URL at runtime — one approach is reading `X-Forwarded-Host` and `X-Forwarded-Proto` from the incoming request headers.
