Every LangSmith deployment exposes a set of HTTP endpoints. Most are *built-in* — the Agent Server API that lets clients manage runs, threads, assistants, and the store. You can also extend the deployment with *custom* endpoints — your own routes mounted alongside the built-in ones. This is how the tutor UI is served directly from the deployment with no separate server.

---

## Built-in routes: the Agent Server API

Every deployment runs the LangSmith Agent Server runtime, which exposes 30+ endpoints grouped roughly as:

| Route group | Purpose |
|---|---|
| `/runs/*` | Start and manage runs |
| `/threads/*` | Manage conversation threads and checkpoints |
| `/assistants/*` | Create, list, update, delete assistants |
| `/store/*` | Read/write key-value memory |
| `/mcp/*` | Model Context Protocol resources |
| `/ok`, `/info`, `/docs`, `/auth/*` | Health, metadata, Swagger, and auth |

You have already used `/runs`, `/threads`, and `/assistants` indirectly via the LangGraph SDK in m2.1 and m2.2 — those SDK calls map to HTTP requests against these built-in routes.

### Browsing all routes

Every deployment also exposes a Swagger UI at `/docs` that lists every available endpoint with request/response schemas — built-in and custom together:

```text
{deployment-url}/docs
```

For local development:

```text
http://localhost:2024/docs
```

For the full schema, see the [Agent Server API reference](https://docs.langchain.com/langsmith/server-api-ref). For Python, the [LangGraph SDK reference](https://langchain-ai.github.io/langgraph/cloud/reference/sdk/python_sdk_ref/) documents the client classes directly.

## Custom routes

The `http.app` field in `langgraph.json` points to a web application object. LangSmith mounts it alongside the built-in routes, so your endpoints live at the same URL as the agent API.

<!-- block: m2.3-langgraph-json -->
<Tabs>
  <Tab title="Python">
```json {6-7}
{
  "dependencies": ["."],
  "graphs": {
    "tutor": "./agent/agent.py:graph"
  },
  "http": {
    "app": "./ui/app.py:app"
  },
  "env": "../.env"
}
```
  </Tab>
  <Tab title="TypeScript">
```json {6-7}
{
  "dependencies": ["."],
  "graphs": {
    "tutor": "./agent/agent.ts:graph"
  },
  "http": {
    "app": "./ui/app.ts:app"
  },
  "env": "../.env"
}
```
  </Tab>
</Tabs>
<!-- /block: m2.3-langgraph-json -->

The docs describe this as "a Starlette app in Python or a Hono app in JavaScript". The tutor server uses FastAPI (Python) and Hono (TypeScript).

---

### The tutor's custom routes

The tutor server adds several endpoints on top of the built-in Agent Server API:

| Route | Method | Purpose |
|---|---|---|
| `/` | GET | Serves `index.html` |
| `/modules` | GET | Returns the module list |
| `/register` | POST | Creates student profile, assistants, and threads |
| `/login` | POST | Looks up an existing student by email |
| `/chat` | POST | Streams a response from the agent |
| `/resources` | GET | Returns all assistants and threads (for debugging) |

These appear in `/docs` alongside the built-in routes.

---

**Calling back into the deployment**

The server uses the LangGraph SDK to forward requests to the agent — which means it needs to know how to talk to the agent. The implementation in the tutor derives the deployment URL from the incoming HTTP request. This allows the server to be used later in the course as a stand-alone server rather than a co-deployed server. This is not necessarily a recommended approach in the general case. The internal loopback client which does not require the URL is an alternate approach.

## Recap

- Every deployment runs the Agent Server API — built-in HTTP endpoints for runs, threads, assistants, store, and more. Visit `{deployment-url}/docs` to browse the full list.
- `http.app` in `langgraph.json` registers a web app as **custom** routes alongside the built-in ones.
- The tutor server adds `/`, `/modules`, `/register`, `/login`, `/chat`, and `/resources`.
- Custom routes that call back into the same deployment need to know how to talk to the agent. The tutor derives the deployment URL from incoming request headers; an internal loopback client is an alternate approach that does not require the URL.
