# The Tutor UI

In this lesson you'll walk through the key pieces of code that connect the tutor UI to your deployed agent. The UI uses a server proxy pattern — a FastAPI (Python) or Hono (TypeScript) server uses the LangGraph SDK to create a client, manage threads, and stream responses, just as you did in the previous lesson.   

The browser talks to the server; the server talks to LangGraph. The server is co-deployed with the agent as a custom route. More on this in the next lesson.

---

## Architecture

```
Browser (index.html)
      |
      | POST /chat
      v
Server (app.py / app.ts, agent_client.py / agentClient.ts)   ← custom route, co-deployed with the agent
      |
      | LangGraph SDK
      v
LangGraph Agent (agent.py / agent.ts)
      |
      | LLM provider
      v
LLM
```

The browser sends messages to the server, which uses the LangGraph SDK to forward requests to the agent. Both run inside the same deployment.

---

## Accessing the UI

The UI is served directly from your deployment. Open your deployment URL in a browser:

```text
https://your-deployment.us.langgraph.app
```

For local development, start the agent with `langgraph dev` and open `http://localhost:2024`.

---

## The Setup Tab

The **setup** tab has two views: **register** (first visit) and **log in** (returning). Register requires your first name, last name, email, and optional course goals. Registering creates one assistant and one thread per module and saves them in `localStorage`. Log in restores an existing session by email.

## Key Code: The Server Proxy

The server (`app.py` / `app.ts`) has two jobs:

1. Serve `index.html`
2. Proxy `/chat` requests to the LangGraph deployment

The browser passes `assistant_id` and `thread_id` directly — both are created at session start and stored in the browser's localStorage.

Called once at registration. Creates one assistant and one thread per module — each pairing tracks a student's conversation and configuration for that module independently.

```typescript {9,12}
export async function createStudentSessions(
  client: Client,
  namespace: string,
  studentName = "Student",
): Promise<Session[]> {
  const sessions: Session[] = [];
  for (const mod of MODULES) {
    const assistant = await client.assistants.create({
      graphId: "tutor",
      name: `${studentName} — ${mod.id}`,
      context: { module_id: mod.id, store_namespace: namespace },
    });
    const thread = await client.threads.create();
    sessions.push({
      module_id: mod.id,
      assistant_id: assistant.assistant_id,
      thread_id: thread.thread_id,
    });
  }
  return sessions;
}
```

### Stream a Response

`stream_mode="messages-tuple"` delivers token deltas as the LLM generates them. Each event is a `(message_chunk, metadata)` tuple — `message_chunk` is the new content, `metadata` has node and invocation details.

Some LLMs return content as a list of typed blocks rather than a plain string, so we normalize that too.

<!-- block: m2.2-stream -->
<Tabs>
  <Tab title="Python">
```python {8-10}
# python/deep_tutor/ui/agent_client.py
async def stream_response(
    client: LangGraphClient,
    thread_id: str,
    assistant_id: str,
    message: str,
) -> AsyncGenerator[str, None]:
    async for event in client.runs.stream(
        thread_id,
        assistant_id,
        input={"messages": [{"role": "user", "content": message}]},
        stream_mode="messages-tuple",
    ):
        if event.event != "messages":
            continue
        message_chunk, _metadata = event.data
        content = message_chunk.get("content", "")
        if isinstance(content, list):
            content = "".join(
                b.get("text", "") for b in content
                if isinstance(b, dict) and b.get("type") == "text"
            )
        if content:
            yield content
```
  </Tab>
  <Tab title="TypeScript">
```typescript {8}
// typescript/deep_tutor/ui/agentClient.ts
async function* streamResponse(
  client: Client,
  threadId: string,
  assistantId: string,
  message: string,
): AsyncGenerator<string> {
  const eventStream = client.runs.stream(threadId, assistantId, {
    input: { messages: [{ role: "user", content: message }] },
    streamMode: "messages-tuple",
  });
  for await (const event of eventStream) {
    if (event.event !== "messages") continue;
    const [messageChunk] = event.data as [Record<string, unknown>, unknown];
    let content = messageChunk.content;
    if (Array.isArray(content)) {
      content = (content as Array<Record<string, unknown>>)
        .filter((b) => b.type === "text")
        .map((b) => b.text ?? "")
        .join("");
    }
    if (typeof content === "string" && content) {
      yield content;
    }
  }
}
```

## Recap

- The UI uses a **server proxy pattern** — the browser talks to a FastAPI (Python) or Hono (TypeScript) server, which uses the LangGraph SDK to forward requests to the agent.
- The server is co-deployed with the agent as a custom route — no separate server to run. m2.3 covers how that works.
- The server derives its own deployment URL from incoming request headers — no configuration needed.
- Registering calls `create_student_sessions` — one assistant and one thread per module, each assistant configured with `module_id` and `store_namespace`.
- The `assistant_id` and `thread_id` are stored in `localStorage` and sent on every `/chat` request — no session state on the server.
- `stream_response` tracks cumulative content per message ID to handle LangGraph's repeated state update events, and normalizes list-format content blocks for LLMs that return them.
