**Summary**

An assistant is a configuration record layered on top of a deployed graph. It lets you:
- Personalize agent behavior at runtime — no redeployment needed
- Create multiple named variants of the same graph, each with its own context
- Store that context server-side — the deployment injects it automatically on every run

---

## Here's the Flow

Here are the steps typically used to connect to a deployed agent:

1. **Get a client** — an SDK handle pointing at the deployment URL with your API key. All subsequent calls (assistants, threads, runs) go through it.
2. **Create an assistant** — a configuration record layered on a deployed graph, as described above. Each graph also has a default assistant created automatically that you can use without creating your own.
3. **Get a thread** — allocates a spot for the conversation in checkpoint memory. This lets thread state persist between runs, even when runs are dispatched to different containers.
4. **Run** — combines an assistant (or the default if none is specified) with a thread to kick off a run.

## Step 3: Create an Assistant

A default assistant is created automatically for each graph when you deploy. Use this step when you want a **named assistant** with custom configuration. For the tutor, you will use the context to create an assistant per course module. That tutor will focus on content related to that module. This is accomplished by passing the `module_id` in the context. We'll also supply the `store_namespace` to identify the storage associated with the student. Recall the agent's dynamic prompt in module 1, lesson 1. It used the `module_id` extracted from context that you are setting here to determine which set of instructions (or SKILL.md files) to insert into the system prompt. Notice that, in this case, the bulk of the data that is used to define the assistant is passed in the skills files in the image, and the context value is used to select the information for a particular assistant. 

You specify:
- `graph_id` — which graph to use
- `name` — a human-readable label
- `context` — configuration (we'll add `module_id` and `store_namespace` in Step 4. It can also be applied here.)

## Step 4: Set the Assistant Context

The agent reads `module_id` and `store_namespace` from the assistant `context` at runtime — it uses `module_id` to load the correct skill files, and `store_namespace` to look up the student's profile from the Store.

Context is stored server-side on the assistant record. You do NOT resend it on every call — the server applies it automatically whenever you use that `assistant_id`.

> **Important:** When updating, include ALL context fields, not just the ones you are changing. The update call replaces the entire context object.

## Step 6a: Run the Assistant

Call `runs.wait()` with the thread and assistant IDs — this sends the message, waits for the full response, and returns the final state.

Two key concepts:
- **Context** (`module_id`, `store_namespace`) is stored on the assistant — you set it once.
- **Config** (e.g., `thread_id`) is passed per run — it identifies the conversation.

<!-- block: m2.1-step6a-code -->
<Tabs>
  <Tab title="Python">
```python
# python/m2/m2.1-connecting.py
result = await client.runs.wait(
    thread["thread_id"],
    assistant_m1["assistant_id"],
    input={"messages": [{"role": "user", "content": "What module is this?"}]},
)
messages = result.get("messages", [])
if messages:
    content = messages[-1].get("content", "")
    if isinstance(content, list):
        content = "".join(
            b.get("text", "") for b in content
            if isinstance(b, dict) and b.get("type") == "text"
        )
    print(content)
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m2/m2.1-connecting.ts
const result = await client.runs.wait(thread.thread_id, assistantM1.assistant_id, {
```

## Step 6b: A Second Assistant — Same Question, Different Context

Now create `assistant_m2` pointing at module 2. Ask the same question — the agent will answer based on the module 2 context, demonstrating that context is what makes each assistant behave differently. This time we stream the response.

<!-- block: m2.1-step6b-code -->
<Tabs>
  <Tab title="Python">
```python
# python/m2/m2.1-connecting.py
assistant_m2 = await client.assistants.create(
    graph_id="tutor",
    name="Tutor — Module 2",
    context={
        "module_id": "module-2",
        "store_namespace": "",
    },
)
print(f"\nCreated assistant: {assistant_m2['assistant_id']}  name={assistant_m2['name']}")

thread_m2 = await client.threads.create()
print(f"Thread: {thread_m2['thread_id']}\n")

async for event in client.runs.stream(
    thread_m2["thread_id"],
    assistant_m2["assistant_id"],
    input={"messages": [{"role": "user", "content": "What module is this?"}]},
    stream_mode="messages-tuple",
):
    if event.event != "messages":
        continue
    message_chunk, _metadata = event.data
    content = message_chunk.get("content", "")
    if isinstance(content, list):
        content = "".join(
            b.get("text", "") for b in content
            if isinstance(b, dict) and b.get("type") == "text"
        )
    if content:
        print(content, end="", flush=True)
print()
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m2/m2.1-connecting.ts
const assistantM2 = await client.assistants.create({
  graphId: "tutor",
  name: "Tutor — Module 2",
  context: {
    module_id: "module-2",
    store_namespace: "",
  },
});
console.log(`\nCreated assistant: ${assistantM2.assistant_id}  name=${assistantM2.name}`);

const threadM2 = await client.threads.create();
console.log(`Thread: ${threadM2.thread_id}\n`);

const stream = client.runs.stream(threadM2.thread_id, assistantM2.assistant_id, {
  input: { messages: [{ role: "user", content: "What module is this?" }] },
  streamMode: "messages-tuple",
});
for await (const event of stream) {
  if (event.event !== "messages") continue;
  const [messageChunk] = event.data as [Record<string, unknown>, unknown];
  let content = messageChunk.content;
  if (Array.isArray(content)) {
    content = (content as Array<Record<string, unknown>>)
      .filter((b) => b.type === "text")
      .map((b) => b.text ?? "")
      .join("");
  }
  if (typeof content === "string" && content) {
    process.stdout.write(content);
  }
}
console.log();
```

## Recap

- **Assistants** are configuration records layered on top of a deployed graph — they hold `module_id`, `store_namespace`, and any other per-assistant settings.
- **Context** (`context` field) is stored server-side on the assistant — set it once, applied automatically on every run.
- **Threads** hold conversation history in checkpoint memory — pass `thread_id` per run to continue the same conversation across runs and containers.
- **Config** (e.g., `thread_id`) is ephemeral — passed per run to identify the conversation thread.
- Two assistants on the same graph behave differently when given different context — no code change required.
- Deleting an assistant frees its config record only — it does not affect compute.
