In the past you may have developed an agent in a notebook or script and run it locally — building the compiled graph in your own process and calling `graph.invoke(...)` or `graph.astream(...)` directly. Now your agent runs inside a container in LangSmith. You don't have the graph object anymore — you talk to it over HTTP using the LangGraph SDK.

By the end of this lesson, you will be able to connect to your deployment and send your first message against it.

---

## Recap: local invoke and stream

When you ran the graph locally, three pieces went into every call:

- **`input`** — the conversation state (messages, etc.)
- **`context`** — typed runtime values matching the graph's `context_schema` (e.g., `module_id`, `store_namespace`)
- **`config`** — LangGraph runtime config: `recursion_limit`, plus `configurable` where `thread_id` lives

<Tabs>
  <Tab title="Python">
```python
result = graph.invoke(
    {"messages": [{"role": "user", "content": "What module is this?"}]},
    context={"module_id": "module-1", "store_namespace": "jane@example_com"},
    config={"configurable": {"thread_id": "abc-123"}},
)
```
  </Tab>
  <Tab title="TypeScript">
```typescript
const result = await graph.invoke(
  { messages: [{ role: "user", content: "What module is this?" }] },
  {
    context: { module_id: "module-1", store_namespace: "jane@example_com" },
    configurable: { thread_id: "abc-123" },
  },
);
```
  </Tab>
</Tabs>

## Same operation, over HTTP

The compiled graph still runs the same way — but it lives in a container now. Your client posts a *run* to the deployment's API; a worker on the deployment side pulls the run from the queue, calls `graph.invoke(...)` internally, and streams the result back.

```text
client.runs.wait(thread_id, assistant_id, input=...)
                                            ↓ HTTP POST /runs
                                  API server enqueues a run
                                            ↓
                                  Queue worker picks it up
                                            ↓
                                  Worker calls graph.invoke(input, config) ← invoke happens HERE
                                            ↓
                                  Returns result via HTTP / SSE
```

Two terms come with this:

- **Assistant** — a configuration record on top of a deployed graph. Every deployed graph auto-creates a *default assistant*; that's what we'll run against in this lesson. Named assistants with custom context are covered in m2.2.
- **Thread** — holds conversation history in the deployment's checkpoint store. Pass `thread_id` on every run against the same conversation.

See the [run execution lifecycle docs](https://docs.langchain.com/langsmith/agent-server#run-execution-lifecycle) for the full lifecycle.

---

## Here's the Flow

Here are the steps to connect to a deployed agent and send your first message:

1. **Get a client** — an SDK handle pointing at the deployment URL with your API key. All subsequent calls (threads, runs) go through it.
2. **Get a thread** — allocates a spot for the conversation in checkpoint memory. This lets thread state persist between runs, even when runs are dispatched to different containers.
3. **Run** — call `runs.wait()` (or `runs.stream()`) with the `thread_id` and an `assistant_id` to send a message. In this lesson the `assistant_id` is just the graph name (`"tutor"`), targeting the default assistant.

## Step 1: Connect to your deployment

Create a client pointing at your deployment URL using your LangSmith API key. The URL defaults to `http://localhost:2024` (local dev); pass your cloud URL as the first argument to target a LangSmith deployment.

The API key is read from your environment — it should never be hardcoded.

## Step 3: Run

Call `runs.wait()` with the thread id and the graph name (`"tutor"`) — this is the deployed equivalent of `graph.invoke()`. It sends the message, waits for the full response, and returns the final state.

<!-- block: m2.1-step3-code -->
<Tabs>
  <Tab title="Python">
```python {4-8}
# python/m2/m2.1-connecting.py
query = "In one sentence, what is the LangGraph runtime?"
print(f"\n→ Running (wait): {query}")
result = await client.runs.wait(
    thread["thread_id"],
    "tutor",
    input={"messages": [{"role": "user", "content": query}]},
)
messages = result.get("messages", [])
if messages:
    content = messages[-1].get("content", "")
    if isinstance(content, list):
        content = "".join(
            b.get("text", "") for b in content
            if isinstance(b, dict) and b.get("type") == "text"
        )
    print(f"✅ Response: {content}")
```
  </Tab>
  <Tab title="TypeScript">
```typescript {4-6}

      .map((b) => b.text ?? "")
      .join("");
  }

}
```

## Step 4: Stream the response

`runs.wait()` waits for the full result before returning. For interactive UIs you want tokens as they're produced — call `runs.stream()` with `stream_mode="messages-tuple"` instead. This is the deployed equivalent of `graph.astream()`.

<!-- block: m2.1-step4-code -->
<Tabs>
  <Tab title="Python">
```python
# python/m2/m2.1-connecting.py
query = "In one sentence, what does the Agent Server include?"
print(f"\n→ Running (stream): {query}")
print("✅ Response: ", end="", flush=True)
async for event in client.runs.stream(
    thread["thread_id"],
    "tutor",
    input={"messages": [{"role": "user", "content": query}]},
    stream_mode="messages-tuple",
):
    if event.event != "messages":
        continue
    message_chunk, _metadata = event.data
    if message_chunk.get("type") != "AIMessageChunk":
        continue  # skip tool calls / tool results — show only the agent's reply
    content = message_chunk.get("content", "")
    if isinstance(content, list):
        content = "".join(
            b.get("text", "") for b in content
            if isinstance(b, dict) and b.get("type") == "text"
        )
    if content:
        print(content, end="", flush=True)
print()
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m2/m2.1-connecting.ts
query = "In one sentence, what does the Agent Server include?";
console.log(`\n→ Running (stream): ${query}`);
process.stdout.write("✅ Response: ");
const stream = client.runs.stream(thread.thread_id, "tutor", {
  input: { messages: [{ role: "user", content: query }] },
  streamMode: "messages-tuple",
});
for await (const event of stream) {
  if (event.event !== "messages") continue;
  const [messageChunk] = event.data as [Record<string, unknown>, unknown];
  if (messageChunk.type !== "ai") continue;  // skip tool calls / tool results — show only the agent's reply
  let content = messageChunk.content;
  if (Array.isArray(content)) {
    content = (content as Array<Record<string, unknown>>)
      .filter((b) => b.type === "text")
      .map((b) => b.text ?? "")
      .join("");
  }
  if (typeof content === "string" && content) {
    process.stdout.write(content);
  }
}
console.log();
```

## Recap

- The graph still runs the same way — but it lives in a container now. You call it via HTTP through the LangGraph SDK.
- `client.runs.wait(thread_id, assistant_id, input=...)` is the deployed equivalent of `graph.invoke(input, config=...)`.
- `client.runs.stream(...)` is the deployed equivalent of `graph.astream(...)`.
- The second argument to `runs.wait`/`runs.stream` is the graph name (`"tutor"`) — every deployed graph is reachable this way without any setup.
- A **thread** holds conversation history in the deployment's checkpoint store; pass `thread_id` per run.

We have omitted an important feature of deployments in LangSmith in this example: Assistants. Runs really take a `thread_id` and an `assistant_id`. In this example, you were given the *default assistant*. To find out more, move to the next lesson.
