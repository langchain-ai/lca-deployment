# Connecting an Assistant to a Deployment

In this lesson, you'll learn how to connect a LangSmith **assistant** to a deployed LangGraph agent — and run your first conversation thread against it.

An assistant is a configuration layer on top of a deployed graph. It lets you:
- Set a custom system prompt
- Configure model and tool settings
- Create multiple named variants of the same graph

By the end of this lesson you will be able to:
1. Create a LangGraph client pointing at a deployment
2. List available graphs
3. Create and configure an assistant
4. Run a streaming conversation thread
5. Clean up by deleting the assistant

---

## Setup

### Environment Variables

As covered in `Getting Started`, your keys live in `./{python,ts}/.env`. For this lesson you will need `LANGSMITH_API_KEY`, `LANGSMITH_PROJECT`, and `LANGGRAPH_URL` to be set.

<!-- block: l2.1-setup-env -->
```bash
# python/.env  (or ts/.env)
LANGSMITH_API_KEY=lsv2_...
LANGSMITH_PROJECT=lca-deployment
LANGGRAPH_URL=http://127.0.0.1:2024   # update to your deployment URL when connecting to LangSmith Deployment
GOOGLE_API_KEY=your-google-api-key
```
<!-- /block: l2.1-setup-env -->

**Reminder — LangSmith Deployment:** update `LANGGRAPH_URL` in `./{python,ts}/.env` with the URL from the LangSmith UI under *Deployments*. Note that when you run `langgraph deploy`, your `.env` is **not** sent to the cloud — set secrets like `GOOGLE_API_KEY` in the LangSmith UI under *Environment Variables and secrets*.

## Step 1: Connect to Your Deployment

Create a client pointing at your deployment URL using your LangSmith API key.

**Finding your URL:**
- **Local (LangGraph Studio):** `http://127.0.0.1:2024`
- **LangSmith Deployment:** find the URL on your deployment's page in the LangSmith UI under *Deployments*

The API key is read from your environment — it should never be hardcoded.

<!-- block: l2.1-step1-code -->
<Tabs>
  <Tab title="Python">
```python
DEPLOYMENT_URL = os.environ.get("LANGGRAPH_URL", "http://127.0.0.1:2024")
API_KEY = os.environ.get("LANGSMITH_API_KEY", "")

client = get_client(url=DEPLOYMENT_URL, api_key=API_KEY)
```
  </Tab>
  <Tab title="TypeScript">
```typescript
const DEPLOYMENT_URL = process.env.LANGGRAPH_URL ?? "http://127.0.0.1:2024";
const API_KEY = process.env.LANGSMITH_API_KEY ?? "";

const client = new Client({ apiUrl: DEPLOYMENT_URL, apiKey: API_KEY });
```

## Step 3: Create an Assistant

A default assistant is created automatically for each graph when you deploy. Use this step when you want a **named assistant** with custom configuration — for example, one per lesson or one per student cohort.

You specify:
- `graph_id` — which graph to use
- `name` — a human-readable label
- `context` — configuration (we'll add `lesson_id` and `student_name` in Step 4)

<!-- block: l2.1-step3-code -->
<Tabs>

## Step 4: Set the Assistant Context

The agent reads `lesson_id` and `student_name` from the assistant `context` at runtime — it uses `lesson_id` to load the correct lesson instructions and material, and `student_name` to personalize responses and look up student data from the store.

Context is stored server-side on the assistant record. You do NOT resend it on every call — the server applies it automatically whenever you use that `assistant_id`.

> **Important:** When updating, include ALL context fields, not just the ones you are changing. The update call replaces the entire context object.

<!-- block: l2.1-step4-code -->
<Tabs>
  <Tab title="Python">
```python
assistant = await client.assistants.update(
    assistant["assistant_id"],
    context={
        "lesson_id": "tutor_l1",
        "student_name": "Student",
    },
)
print(f"Updated assistant: {assistant['assistant_id']}")
```
  </Tab>
  <Tab title="TypeScript">
```typescript
assistant = await client.assistants.update(assistant.assistant_id, {
  context: {
    lesson_id: "tutor_l1",
    student_name: "Student",
  },
});
console.log(`Updated assistant: ${assistant.assistant_id}`);
```

## Step 5: Run a Thread Using the Assistant

A **thread** holds the conversation history. Create a thread, then stream a run against it using your assistant.

Two key concepts:
- **Context** (`lesson_id`, `student_name`) is stored on the assistant — you set it once.
- **Config** (e.g., `thread_id`) is passed per run — it identifies the conversation.

The stream returns events. We look for messages with `type: "ai"` to extract the agent's text response.

<!-- block: l2.1-step5-code -->
<Tabs>
  <Tab title="Python">
```python
thread = await client.threads.create()
print(f"\nThread: {thread['thread_id']}")

printed = 0  # track cumulative content already printed
async for event in client.runs.stream(
    thread["thread_id"],
    assistant["assistant_id"],
    input={"messages": [{"role": "user", "content": "What is the Agent Server?"}]},
    stream_mode="messages",
):
    if isinstance(event.data, list):
        for item in event.data:
            msg = item[0] if isinstance(item, (list, tuple)) else item
            if isinstance(msg, dict) and msg.get("type") in ("AIMessageChunk", "AIMessage", "ai"):
                content = msg.get("content", "")
                if isinstance(content, str) and len(content) > printed:
                    print(content[printed:], end="", flush=True)
                    printed = len(content)
print()
```
  </Tab>
  <Tab title="TypeScript">
```typescript
const thread = await client.threads.create();
console.log(`\nThread: ${thread.thread_id}`);

let printed = 0; // track cumulative content already printed
const stream = client.runs.stream(thread.thread_id, assistant.assistant_id, {
  input: { messages: [{ role: "user", content: "What is the Agent Server?" }] },
  streamMode: "messages",
});

for await (const event of stream) {
  if (Array.isArray(event.data)) {
    for (const item of event.data as Array<unknown>) {
      const msg = Array.isArray(item) ? item[0] : item;
      if (msg && typeof msg === "object") {
        const m = msg as Record<string, unknown>;
        if (m.type === "AIMessageChunk" || m.type === "AIMessage" || m.type === "ai") {
          if (typeof m.content === "string" && m.content.length > printed) {
            process.stdout.write(m.content.slice(printed));
            printed = m.content.length;
          }
        }
      }
    }
  }
}
console.log();
```

</Tabs>

```text
Thread: 1166288c-98a2-404b-84db-6a1c772624cd

That's a great question!

The Agent Server is the complete deployment instance for your LangGraph agents. It's
not a single component, but rather a collection of infrastructure that runs your agents
in production. This includes:

- API servers: handle incoming requests from clients
- Worker containers: execution engines that run your agent's graph code
- Postgres: durable source of truth for threads, runs, and checkpoints
- Redis: ephemeral signaling layer for waking workers and streaming output

It's important to note that the Agent Server is not truly serverless. The worker
containers are always-on and consume resources even when idle, ensuring there are no
cold starts when a request comes in.
```

## Step 6: Delete the Assistant

Deleting an assistant frees the configuration record. It does **not** affect running containers or compute resources — assistants are config only.

<!-- block: l2.1-step6-code -->
<Tabs>
  <Tab title="Python">
```python

print(f"\nDeleted assistant: {assistant['assistant_id']}")
```

## Recap

- **Assistants** are configuration records layered on top of a deployed graph — they hold `lesson_id`, `student_name`, and any other per-assistant settings.
- **Context** (`context` field) is stored server-side on the assistant — set it once, applied automatically on every run.
- **Config** (e.g., `thread_id`) is ephemeral — passed per run to identify the conversation thread.
- Deleting an assistant frees its config record only — it does not affect compute.
