**Summary**

An assistant is a configuration record layered on top of a deployed graph. It lets you:
- Personalize agent behavior at runtime — no redeployment needed
- Create multiple named variants of the same graph, each with its own context
- Store that context server-side — the deployment injects it automatically on every run

---

## Here's the Flow

Here are the steps to create and run a named assistant:

1. **Get a client** — same as m2.1, an SDK handle pointing at your deployment.
2. **Create a named assistant** — call `client.assistants.create()` with the `graph_id`, a `name`, and a `context`. The context is stored server-side and injected into every run that uses this assistant.
3. **Get a thread** — a conversation slot in checkpoint memory, same as m2.1.
4. **Run** — call `runs.wait()` (or `runs.stream()`) with the thread id and the new `assistant_id`. The agent reads the bound context and behaves accordingly.

## Step 2: Create a named assistant

Call `client.assistants.create()` with the `graph_id`, a `name`, and a `context`. The context is typed runtime values matching the graph's `context_schema`. For the tutor, the agent reads `module_id` to load the right teaching skills and `store_namespace` to look up the student's stored profile.

Context is stored server-side on the assistant record. You do **not** resend it on every run — the server applies it automatically whenever you use that `assistant_id`.

<!-- block: m2.2-step2-code -->
<Tabs>
  <Tab title="Python">
```python
# python/m2/m2.2-assistants.py
assistant_m1 = await client.assistants.create(
    graph_id="tutor",
    name="Tutor — Module 1",
    context={
        "module_id": "module-1",
        "store_namespace": "",
    },
)
print(f"✅ Created assistant_m1: {assistant_m1['assistant_id']}  name={assistant_m1['name']}")
```

## Step 4: Run

Same `runs.wait()` call as m2.1, but pass the named assistant's id instead of the graph name. The server pulls the bound context and the agent answers from the module-1 perspective.

<!-- block: m2.2-step4-code -->
<Tabs>
  <Tab title="Python">
```python
# python/m2/m2.2-assistants.py
query = "In one sentence, what module is this?"
print(f"\n→ Running (wait, assistant_m1): {query}")
result = await client.runs.wait(
    thread["thread_id"],
    assistant_m1["assistant_id"],
    input={"messages": [{"role": "user", "content": query}]},
)
messages = result.get("messages", [])
if messages:
    content = messages[-1].get("content", "")
    if isinstance(content, list):
        content = "".join(
            b.get("text", "") for b in content
            if isinstance(b, dict) and b.get("type") == "text"
        )
    print(f"✅ Response: {content}")
```

## Step 5: Same question, different context

Create a second assistant with `module_id="module-2"`, give it its own thread, and send the same question — streamed this time. The agent answers from the module-2 perspective. The only thing that changed is the context bound to each assistant.

<!-- block: m2.2-step5-code -->
<Tabs>
  <Tab title="Python">
```python
# python/m2/m2.2-assistants.py
assistant_m2 = await client.assistants.create(
    graph_id="tutor",
    name="Tutor — Module 2",
    context={
        "module_id": "module-2",
        "store_namespace": "",
    },
)
print(f"\n✅ Created assistant_m2: {assistant_m2['assistant_id']}  name={assistant_m2['name']}")

thread_m2 = await client.threads.create()
print(f"✅ Thread created: {thread_m2['thread_id']}")

print(f"\n→ Running (stream, assistant_m2): {query}")
print("✅ Response: ", end="", flush=True)
async for event in client.runs.stream(
    thread_m2["thread_id"],
    assistant_m2["assistant_id"],
    input={"messages": [{"role": "user", "content": query}]},
    stream_mode="messages-tuple",
):
    if event.event != "messages":
        continue
    message_chunk, _metadata = event.data
    if message_chunk.get("type") != "AIMessageChunk":
        continue  # skip tool calls / tool results — show only the agent's reply
    content = message_chunk.get("content", "")
    if isinstance(content, list):
        content = "".join(
            b.get("text", "") for b in content
            if isinstance(b, dict) and b.get("type") == "text"
        )
    if content:
        print(content, end="", flush=True)
print()
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m2/m2.2-assistants.ts
const assistantM2 = await client.assistants.create({
  graphId: "tutor",
  name: "Tutor — Module 2",
  context: {
    module_id: "module-2",
    store_namespace: "",
  },
});
console.log(`\n✅ Created assistantM2: ${assistantM2.assistant_id}  name=${assistantM2.name}`);

const threadM2 = await client.threads.create();
console.log(`✅ Thread created: ${threadM2.thread_id}`);

console.log(`\n→ Running (stream, assistantM2): ${query}`);

  input: { messages: [{ role: "user", content: query }] },
  streamMode: "messages-tuple",
});
for await (const event of stream) {
  if (event.event !== "messages") continue;
  const [messageChunk] = event.data as [Record<string, unknown>, unknown];
  if (messageChunk.type !== "AIMessageChunk") continue;  // skip tool calls / tool results — show only the agent's reply

  if (Array.isArray(content)) {
    content = (content as Array<Record<string, unknown>>)
      .filter((b) => b.type === "text")
      .map((b) => b.text ?? "")
      .join("");
  }
  if (typeof content === "string" && content) {
    process.stdout.write(content);
  }
}
console.log();
```

## Sidebar: context per-run

Storing context on the assistant is the common case. For one-off variations, you can also pass `context` directly to `runs.wait()` or `runs.stream()` — the values are used for that run only and don't change what's stored on the assistant:

<Tabs>
  <Tab title="Python">
```python
result = await client.runs.wait(
    thread["thread_id"],
    assistant_m1["assistant_id"],
    input={"messages": [{"role": "user", "content": "What module is this?"}]},
    context={"module_id": "module-2", "store_namespace": ""},   # overrides for this call only
)
```
  </Tab>
  <Tab title="TypeScript">
```typescript
const result = await client.runs.wait(thread.thread_id, assistantM1.assistant_id, {
  input: { messages: [{ role: "user", content: "What module is this?" }] },
  context: { module_id: "module-2", store_namespace: "" },   // overrides for this call only
});
```

## Recap

- **Assistants** are configuration records layered on top of a deployed graph — each holds a `context` value used at runtime.
- `client.assistants.create(graph_id=..., name=..., context=...)` makes a named variant in one call.
- **Context** is stored server-side on the assistant — set it once, applied automatically on every run.
- `client.assistants.update(...)` replaces the entire context object — include all fields, not just the ones changing.
- Two assistants on the same graph behave differently when given different context — no code change required.
- Deleting an assistant frees the config record only — it does not affect compute.
