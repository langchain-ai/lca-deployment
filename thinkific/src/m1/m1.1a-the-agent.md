# The Tutor Agent

The agent you are about to deploy is a tutor. It teaches LangChain, LangGraph, and LangSmith — the same tools you are using to build and deploy it. The code itself is the subject of the course.

The key idea in this lesson: **the agent code is fixed at deploy time, but its behavior is shaped at runtime.** A single deployed agent can act as a different tutor for each student and each lesson — because the context schema and dynamic prompt middleware let you inject configuration into every run without changing the code.

This lesson walks through `agent.py` with that idea in mind.

---

## The Files

The agent lives in `lang_chat/agent/`. Lesson content lives alongside it in `tutor_l1/`.

```
lang_chat/
  agent/
    agent.py / agent.ts            ← the agent (this lesson)
    system_prompt.py / systemPrompt.ts   ← base tutor personality
  tutor_l1/
    tutor_l1_instructions.md       ← what to teach in lesson 1
    tutor_l1_information.md        ← reference material the agent can read
```

The agent file is the only file LangGraph deploys — it imports the system prompt and reads the `tutor_l1/` files at runtime.

---

## The Agent: `create_agent`

The entire agent is defined by a single call to `create_agent`. This builds a complete tool-calling loop — the model calls tools, gets results, and keeps going until it has a final answer.

The four parameters are:

- **`model`** — the LLM to use. Set once, used for every turn.
- **`tools`** — functions the agent can call. Here: read lesson material, and search the LangChain docs via MCP.
- **`middleware`** — runs before each model call. Used here to inject the dynamic system prompt.
- **`context_schema`** — defines the per-assistant configuration the agent can read.

    model=model,

## Context Schema

The context schema defines what configuration each assistant can carry. When you create an assistant in LangSmith and set its context dict, the values are validated against this schema and made available to the agent at runtime.

The tutor uses three fields:

- **`lesson_id`** — which lesson to load (e.g. `"tutor_l1"`). Controls which instruction and material files are read.
- **`student_name`** — the student's first name, used to personalize responses.
- **`store_namespace`** — a key for the student's persistent data in the LangGraph store (e.g. `"jane_doe"`).

Each field has a default, so the agent works even if context is not set — but it will use `"Student"` as the name and load lesson 1.

## Dynamic Prompt Middleware

Middleware runs before each model call. The `@dynamic_prompt` decorator marks a function that returns the system prompt — LangGraph calls it every turn, passing a `request` object that includes the runtime context.

This is how the agent knows which student it is talking to and which lesson it is teaching: the context fields are read from `request.runtime.context` and injected into the system prompt on every call.

The result is a prompt that is fully assembled at runtime — never hardcoded.

<!-- block: m1.1a-middleware -->
<Tabs>
  <Tab title="Python">
```python
@dynamic_prompt
def lesson_prompt(request: ModelRequest) -> str:
    lesson_id = request.runtime.context.lesson_id
    student_name = request.runtime.context.student_name
    instructions = _load_lesson_instructions(lesson_id)
    return (
        f"{BASE_SYSTEM_PROMPT}\n\n"
        f"## Current session\n"
        f"Student: {student_name}\n"
        f"Lesson: {lesson_id}\n\n"
        f"## Lesson instructions\n\n"
        f"{instructions}"
    )
```
  </Tab>
  <Tab title="TypeScript">
```typescript
const lessonPrompt = dynamicSystemPromptMiddleware<Context>(
  (state, runtime) => {
    const lessonId = runtime.context.lessonId;
    const studentName = runtime.context.studentName;
    const instructions = loadLessonInstructions(lessonId);
    return (
      `${BASE_SYSTEM_PROMPT}\n\n` +
      `## Current session\n` +
      `Student: ${studentName}\n` +
      `Lesson: ${lessonId}\n\n` +
      `## Lesson instructions\n\n` +
      instructions
    );
  }
);
```

## Giving the Agent a Personality

The system prompt the middleware assembles has three parts:

**1. Base persona** — defined in `system_prompt.py` / `systemPrompt.ts`. This is the tutor's core character: warm, concise, uses the lesson material first, occasionally quizzes the student.

<!-- block: m1.1a-base-persona -->
<Tabs>
  <Tab title="Python">
```python
BASE_SYSTEM_PROMPT = """You are an empathetic but concise tutor helping students
learn LangChain, LangGraph, and LangSmith.

## Your approach
- Be warm and encouraging, but keep explanations focused and to the point
- When answering a question, use this priority order:
  1. The lesson instructions provided in this prompt
  2. The lesson material — use the read_lesson_material tool to retrieve it
  3. The LangChain documentation — use the MCP tools to search if needed
- Occasionally quiz the student to reinforce learning
- If a student seems confused, slow down and use an analogy
- If they're moving fast, keep up with them
"""
```

**2. Lesson instructions** — loaded from `tutor_l1/tutor_l1_instructions.md` at runtime. These tell the agent *what* to teach in this lesson: the key concepts, how to run the lesson, and the tone to use. Each lesson has its own instructions file.

**3. Lesson material** — the agent does not receive the full reference material in its prompt. Instead, it has a `read_lesson_material` tool it calls when it needs to answer a question or quiz the student. This keeps the base prompt lean.

<!-- block: m1.1a-tool -->
<Tabs>
  <Tab title="Python">
```python
@tool
def read_lesson_material(runtime: ToolRuntime[ContextSchema]) -> str:
    """Read the reference material for the current lesson.

    Use this tool when you need to answer a student's question or quiz them.
    Always check the lesson material before answering.
    """
    lesson_id = runtime.context.lesson_id
    info_file = LESSONS_DIR / lesson_id / f"{lesson_id}_information.md"
    if not info_file.exists():
        return f"Lesson material not found for '{lesson_id}'."
    return info_file.read_text()
```
  </Tab>
  <Tab title="TypeScript">
```typescript
const readLessonMaterial = tool(
  async (_: object, runtime: ToolRuntime<Context>) => {
    const lessonId = runtime.context.lessonId;
    const infoFile = path.join(LESSONS_DIR, lessonId, `${lessonId}_information.md`);
    if (!fs.existsSync(infoFile)) {
      return `Lesson material not found for '${lessonId}'.`;
    }
    return fs.readFileSync(infoFile, "utf-8");
  },
  {
    name: "read_lesson_material",
    description:
      "Read the reference material for the current lesson. " +
      "Use this tool when you need to answer a student's question or quiz them. " +
      "Always check the lesson material before answering.",
    schema: z.object({}),
  }
);
```

</Tabs>

## Recap

The agent code is deployed once. Its behavior varies at runtime:

- **`context_schema`** defines the per-assistant configuration — `lesson_id`, `student_name`, `store_namespace`. You set these values when you create an assistant in LangSmith; they are injected automatically on every run.
- **Dynamic prompt middleware** reads those values before each model call and assembles the system prompt — different student, different lesson, different prompt. No code change required.
- The agent's personality comes from three layered sources: base persona (always on), lesson instructions (per lesson, loaded from file), and lesson material (on demand via tool).
- `create_agent` wires all of this together: model + tools + middleware + context schema.
