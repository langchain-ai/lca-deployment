The agent you are about to deploy is a tutor — a LangChain Academy coach that teaches the very course you are taking. Once deployed, you can use it as a study aid and ask it questions about deployment as you work through the modules.

The agent is built with the deepagents library (<a href="https://docs.langchain.com/oss/python/deepagents/overview">docs</a>, <a href="https://pypi.org/project/deepagents/">Python package</a>, <a href="https://www.npmjs.com/package/deepagents">npm package</a>), which provides a structured way to build production-ready agents with built-in memory, skills, and file access. This course is about deployment rather than creating agents, but this lesson walks through how it works for completeness and for the curious. If you are pressed for time, skim or skip to the next lesson.

One note, it is hard to resist adding features to the tutor over time. It's possible the code in the git repo may be slightly different than what is described!

## The Files

The agent lives in `deep_tutor/agent/`. Teaching content is in `skills/`. The agent's personality is defined in `AGENTS.md`. The use of skills and `AGENTS.md` has become a convention in agent definition. It is supported with middleware. This course won't describe this in detail, but you can find more details about `AGENT.md`[here](https://docs.langchain.com/oss/python/deepagents/cli/memory-and-skills#agents-md-files) and more information on skills [here](https://docs.langchain.com/oss/python/deepagents/skills).

## The DeepAgent

This agent uses the deepagents library. By following a few conventions — where to put and name memory files, and how to structure skills — the agent gains memory, progressive skill loading, and filesystem access.

## Context Schema

The context schema defines what per-assistant configuration the agent can read at runtime. When you create an assistant in LangSmith and set its context values, those values are validated against this schema and passed to the agent on every run.

The tutor uses two fields:

- **`module_id`** — which module to teach (e.g. `"module-1"`). Controls which skill files are loaded.
- **`store_namespace`** — the student's namespace in the LangGraph Store, derived from their email (e.g. `"jane@example_com"`). The agent reads the student's name and goals from the Store using this key.

<!-- block: m1.1-context-schema -->
<Tabs>
  <Tab title="Python">
```python
# python/deep_tutor/agent/agent.py

    module_id: str = "module-1"
    store_namespace: str = ""  # derived from student email
```

## Memory

The `memory` parameter takes a list of file paths. deepagents loads those files on every run and prepends their content to the system prompt before the first model call.

For the tutor, that file is `AGENTS.md`:

```markdown
# Agent Instructions

You are an empathetic but concise tutor for the LangChain Academy deployment course.
You teach students how to deploy LangGraph agents to production — covering architecture,
storage, auth, and the full deployment lifecycle.
...
```

Unlike skills, memory is not tied to any module — it is the agent's permanent personality and applies to every session.

---

## Skills

The `skills` parameter points to a directory. deepagents uses progressive disclosure: at startup it indexes all available skills; it then loads the content of individual skills on demand as the conversation evolves.

Each skill lives in its own subdirectory and contains two files:

<!-- block: m1.1-skills-tree -->
```text
skills/
  module-1/
    SKILL.md          ← how to teach the module (key concepts, tone, quiz guidance)
    information.md    ← reference material the agent reads when answering questions
  module-n/
    SKILL.md
    information.md
```
<!-- /block: m1.1-skills-tree -->

`SKILL.md` is the teaching plan: which concepts to cover, in what order, and when to quiz. `information.md` is the reference — the agent reads it on demand rather than having it in the prompt at all times.

The dynamic prompt (next section) tells the agent to read the right `SKILL.md` on its first response.

## The Filesystem Backend

Skills and memory are loaded from files via `FilesystemBackend`. The backend exposes a virtual filesystem to the agent — all file paths the agent uses are relative to the project root, regardless of where the process is running.

`virtual_mode=True` / `virtualMode: true` resolves paths on disk but presents them to the agent as absolute paths like `/AGENTS.md` or `/skills/module-1/SKILL.md`.

Permissions restrict what the agent can do with its files. The tutor can read everything but cannot write to the skills directory:

<!-- block: m1.1-backend -->
<Tabs>
  <Tab title="Python">
```python
# python/deep_tutor/agent/agent.py
backend = FilesystemBackend(
    root_dir=Path(__file__).parent.parent,
    virtual_mode=True,
)

# passed to create_deep_agent:
permissions=[
    FilesystemPermission(operations=["write"], paths=["/skills/**"], mode="deny"),
],
```
  </Tab>

## Dynamic Prompt

Middleware runs before each model call. The tutor's middleware reads the context values and assembles the session-specific portion of the system prompt: which student is in the session, which module they are working on, and what goals they have.

When a `store_namespace` is set, it also reads the student's profile from the LangGraph Store, overriding the context values for name and goals.

<!-- block: m1.1-middleware -->
<Tabs>
  <Tab title="Python">
```python
# python/deep_tutor/agent/agent.py
@dynamic_prompt
async def tutor_prompt(request: ModelRequest) -> str:
    module_id = request.runtime.context.module_id
    store_namespace = request.runtime.context.store_namespace
    student_name = "Student"
    goals = ""

    if store_namespace and request.runtime.store:
        item = await request.runtime.store.aget((store_namespace,), key="profile")
        if item is not None:
            profile = item.value
            student_name = (
                f"{profile.get('first_name', '')} {profile.get('last_name', '')}".strip()
                or "Student"
            )
            goals = profile.get("goals", "")

    goals_line = f"\n\nThe student's goals: {goals}" if goals else ""
    return (
        f"Your student's name is {student_name}. "
        f"They are working on {module_id}.{goals_line}\n\n"
        f"On your first response, read /skills/{module_id}/SKILL.md. "
        f"That file contains your teaching instructions — follow them. "
        f"When you need to answer a factual question, read /skills/{module_id}/information.md."
    )
```
  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/deep_tutor/agent/agent.ts
const tutorPrompt = createMiddleware({
  name: "TutorPrompt",

  async wrapModelCall(request: any, handler: any) {
    const configurable = (getConfig().configurable ?? {}) as Record<string, string>;
    const moduleId = configurable.module_id ?? "module-1";
    const storeNamespace = configurable.store_namespace ?? "";
    let studentName = "Student";
    let goals = "";

    if (storeNamespace) {
      const store = getStore();
      if (store) {
        const item = await store.aget([storeNamespace], "profile");
        if (item) {
          const profile = item.value as Record<string, string>;
          const fullName = `${profile.first_name ?? ""} ${profile.last_name ?? ""}`.trim();
          if (fullName) studentName = fullName;
          if (profile.goals) goals = profile.goals;
        }
      }
    }

    const goalsLine = goals ? `\n\nThe student's goals: ${goals}` : "";
    const promptText =
      `Your student's name is ${studentName}. ` +
      `They are working on ${moduleId}.${goalsLine}\n\n` +
      `On your first response, read /skills/${moduleId}/SKILL.md. ` +
      `That file contains your teaching instructions — follow them. ` +
      `When you need to answer a factual question, read /skills/${moduleId}/information.md.`;

    return handler({
      ...request,
      systemMessage: request.systemMessage.concat(promptText),
    });
  },
});
```
  </Tab>
</Tabs>
<!-- /block: m1.1-middleware -->

The TypeScript version uses `createMiddleware` + `wrapModelCall` and *appends* to the existing `systemMessage` rather than replacing it. deepagents builds the system prompt in layers — memory, skills, and then middleware. Replacing it would discard the memory and skills layers. The Python `@dynamic_prompt` decorator handles this automatically.

## MCP Tools

The agent has access to the LangChain documentation via an MCP server. `MultiServerMCPClient` fetches tool definitions from the server at startup; the agent calls those tools like any other when it needs to answer a question not covered by its skill files.

## Recap

The agent code is deployed once. Its behavior varies at runtime:

- **Context schema** defines per-assistant configuration — `module_id` and `store_namespace`. These values are set when you create an assistant in LangSmith and passed to the agent on every run.
- **Memory** loads `AGENTS.md` into the system prompt on every run — the agent's personality and teaching style.
- **Skills** are loaded progressively from `/skills/`. Each module has a `SKILL.md` (teaching plan) and an `information.md` (reference material read on demand).
- **FilesystemBackend** exposes files as a virtual filesystem with configurable permissions. The agent can read anything but cannot write to the skills directory.
- **Dynamic prompt middleware** reads the context before each model call and assembles the session-specific part of the prompt — which student, which module, what goals.
- **`create_deep_agent` / `createDeepAgent`** takes the model, tools, middleware, backend, and schema and returns a LangGraph graph.
