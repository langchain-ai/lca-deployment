# Deploying the Agent

`langgraph deploy` builds a Docker container and pushes it to LangSmith. Three things go into that container: your **code**, your **dependencies**, and your **runtime environment**. `langgraph.json` tells LangGraph where to find all three. The `langgraph.json` file can also configure other aspects of your deployment such as assistants, checkpointers, api servers and more.

---

## `langgraph.json`

This is the deployment manifest. It lives at the root of your project and is the only configuration file LangGraph needs to build and deploy your agent.

<!-- block: m1.2-langgraph-json -->
<Tabs>
  <Tab title="Python">
```json
{
  "dependencies": ["."],
  "graphs": {
    "tutor": "./agent/agent.py:graph"
  },
  "http": {
    "app": "./ui/app.py:app"
  },
  "env": ".env"
}
```
  </Tab>
  <Tab title="TypeScript">
```json
{
  "dependencies": ["."],
  "graphs": {
    "tutor": "./agent/agent.ts:graph"
  },
  "http": {
    "app": "./ui/app.ts:app"
  },
  "env": ".env"
}
```
  </Tab>
</Tabs>
<!-- /block: m1.2-langgraph-json -->

**`graphs` — the code**

Maps a graph name to the source file and exported variable. This is what gets deployed as your agent. The name `"tutor"` is what you will see in LangSmith and what you pass as `graph_id` when creating assistants.

**`http.app` — custom routes**

Points to a FastAPI (Python) or Hono (TypeScript) app that gets served alongside the agent. This is how the tutor UI is deployed — no separate server needed.

**`dependencies` — the dependencies**

Points to the directory containing your package manifest (`pyproject.toml` for Python, `package.json` for TypeScript). LangGraph installs these into the container at build time.

Both languages use `["."]` — the package manifest lives in the same directory as `langgraph.json`. You may notice a second package manifest one level up (`python/pyproject.toml` or `typescript/package.json`). That one is for local development only — running lesson scripts and the dev server. The deployment only needs a subset of those packages, so we keep them separate.

<Tabs>
  <Tab title="Python">
```text
python/
  pyproject.toml          ← local dev: lesson scripts, langgraph-cli, etc.
  deep_tutor/
    pyproject.toml        ← deployment: agent deps + UI server only
    langgraph.json
    agent/
    ui/
```
  </Tab>
  <Tab title="TypeScript">
```text
typescript/
  package.json            ← local dev: lesson scripts
  deep_tutor/
    package.json          ← deployment: agent deps + UI server
    langgraph.json
    agent/
    ui/
```
  </Tab>
</Tabs>

**`env` — the environment**

Path to a `.env` file. When you run `langgraph dev`, these values are loaded into the local process. When you run `langgraph deploy`, the non-reserved values are sent to the deployment — variables like `LANGSMITH_API_KEY` and `LANGSMITH_PROJECT` are skipped automatically since LangSmith manages those.

### Keys

You need two kinds of keys:

- **`LANGSMITH_API_KEY`** — authenticates with LangSmith for tracing, the deployment API, and the Studio UI
- **`ANTHROPIC_API_KEY`** (or your LLM provider's key) — used by the agent to call the model

Both live in your `.env` file. `LANGSMITH_API_KEY` is used locally and in the cloud (managed by LangSmith). `ANTHROPIC_API_KEY` is sent to the deployment via the `env` field — you can also set or override it in the LangSmith UI after deploying.

## The Full `langgraph.json` Schema

`langgraph.json` defines every aspect of how your agent(s) are built, deployed, and run — from the container OS and Python version, to the checkpointer backend, to which API endpoints are enabled. Many of these have default values which are sufficient for most applications, but a closer look at the options provides additional insight into deployments.

Full reference: [langgraph.json configuration file](https://docs.langchain.com/langsmith/cli#configuration-file)

<details>
<summary>🐳 Docker image — container build</summary>

| Field | Description |
|---|---|
| `dependencies` | Packages installed into the image (`pyproject.toml`, `requirements.txt`, or inline list) |
| `base_image` | Pins the LangGraph runtime version |
| `image_distro` | Container OS — `debian` (default) or `wolfi` |
| `python_version` | `3.11`, `3.12`, or `3.13` |
| `node_version` | For JS deployments |
| `pip_installer` | `pip` or `uv` |
| `pip_config_file` | Path to a `pip.conf` or `uv.toml` |
| `keep_pkg_tools` | Keep pip/uv in the final image (default: removed) |
| `dockerfile_lines` | Custom Dockerfile additions (e.g. `apt-get install`) |

</details>

<details>
<summary>🤖 Assistants</summary>

| Field | Description |
|---|---|
| `graphs` | Maps graph ID → code path (`"tutor": "./agent/agent.py:graph"`). One entry per graph. |

</details>

<details>
<summary>💾 Checkpointer</summary>

| Field | Description |
|---|---|
| `checkpointer.backend` | `postgres` (default), `mongo`, or `custom` |
| `checkpointer.path` | Path to custom checkpointer factory |
| `checkpointer.ttl.strategy` | Expiry strategy |
| `checkpointer.ttl.interval` | How often TTL sweep runs |
| `checkpointer.ttl.default_ttl` | Default lifespan for checkpoints |
| `checkpointer.serde` | Deserialization behavior |

</details>

<details>
<summary>🗄️ Store</summary>

| Field | Description |
|---|---|
| `store.index.embed` | Embedding model for semantic search |
| `store.index.dims` | Embedding dimensions |
| `store.index.fields` | Which fields to index |
| `store.ttl.refresh_on_read` | Reset TTL when an item is read |
| `store.ttl.default_ttl` | Default item lifespan |
| `store.ttl.sweep_interval` | How often expired items are swept |

</details>

<details>
<summary>🌐 API Server</summary>

| Field | Description |
|---|---|
| `http.app` | Custom routes — FastAPI/Starlette (Python) or Hono (TypeScript) |
| `http.cors` | Allowed origins, methods, headers |
| `http.middleware_order` | Order middleware is applied |
| `http.configurable_headers` / `logging_headers` | Headers passed through or logged |
| `http.mount_prefix` | URL prefix for all routes |
| `http.disable_*` | Disable built-in endpoints: `assistants`, `mcp`, `runs`, `threads`, `store`, etc. |
| `auth` | Path to custom auth handler |
| `webhooks` | Outbound webhook headers and URL policy |

</details>

<details>
<summary>🔧 Cross-cutting</summary>

| Field | Description |
|---|---|
| `env` | Path to `.env` file — values loaded locally; non-reserved values sent to deployment |
| `api_version` | Pins the Agent Server API version |

</details>

## Run Locally: `langgraph dev`

Before deploying, verify the agent works on your machine. `langgraph dev` runs the deployment locally with hot-reloading. It also launches LangGraph Studio, which you can use to inspect the graph, create threads, and debug the agent before pushing to the cloud.

## Deploy: `langgraph deploy`

Once you are satisfied locally, deploy to LangSmith. The `langgraph` CLI authenticates with LangSmith using `LANGSMITH_API_KEY` — it looks for it in the `.env` file configured in `langgraph.json` first, then in your shell environment. If neither has it, the CLI will prompt you to enter it interactively.

Run this from the same directory as `langgraph.json`:

<Tabs>
  <Tab title="Python">
```bash
cd lca-deployment/python/deep_tutor
uv run langgraph deploy
```
  </Tab>
  <Tab title="TypeScript">
```bash
cd lca-deployment/typescript/deep_tutor
npx @langchain/langgraph-cli deploy
```
  </Tab>
</Tabs>

This builds the Docker container, pushes it to LangSmith, and starts the deployment. The first build takes a few minutes. Example output:

## The Deployments Tab

In LangSmith, open **Deployments** from the left nav. You will see your deployment with a build status indicator. Once it turns green the agent is live.

From this page you can:

- **Find your deployment URL** — you will need this in the next lesson to connect the UI
- **Open Studio against the cloud deployment** — same Studio interface you used locally, but now running against the live container. Use this to verify the deployed agent behaves the same as it did locally before connecting the real UI.
- **Manage environment variables and secrets** — add or update your `LANGSMITH_API_KEY` and `ANTHROPIC_API_KEY` here

---

## Recap

- `langgraph.json` defines what goes into the container: code (`graphs`), dependencies (`dependencies`), and local env (`env`).
- Keys for local dev go in `.env`. Keys for cloud go in the LangSmith UI — never in source code.
- `langgraph dev` runs the agent locally; verify it in Studio before deploying.
- `langgraph deploy` builds and pushes the container. Check the Deployments tab for build status and your deployment URL.
- Studio works against the cloud deployment too — use it to confirm the live agent before connecting the UI.

In the next lesson you will connect the tutor UI to your deployment.
