As your agent moves to production, it will likely be run by many users. Naturally, in a Multi-tenant situation like this, you would like to 1) understand who is using your system and 2) specify resources and actions that are allowed to that user. These are referred to as authentication and authorization. The combination is often referred to as 'auth'.

<hr style="border: none; border-top: 1px solid #1E3A5F; width: 66%; margin: 24px auto;" />

There are three applications of Authentication and Authorization to consider: 

- **Controlling user access:** This relates to your users and is the focus of this module.
- **Controlling agent access:** This relates to the access provided to a user's agents. For example, do they have access to the user's email so they can read emails on the user's behalf? This is "Agent Auth"; you can find more details [here](https://docs.langchain.com/langsmith/agent-auth). 
- **Controlling the deployment:** This relates to you and your team. LangSmith [RBAC (Role-Based Access Control)](https://docs.langchain.com/langsmith/rbac) gives you control over who can deploy, update, and monitor your agents.

## LangSmith Auth

You have been using auth throughout this course! You have created an account on LangSmith and established a password. When you log in to LangSmith, you establish your identity. Once logged in, you created a unique API key that can later be used to identify you in LangSmith. 

Every time you call one of your deployment endpoints, you include your LangSmith API key in the request. That key drives a full authn + authz flow: LangSmith authenticates the key (identifying the LangSmith user it belongs to) and authorizes the request against the resources that user has access to — this specific deployment, among others. A different user's key would not work here.
For example, you may request an agent run using the SDK:

```python
client = get_client(url=DEPLOYMENT_URL, api_key=LANGSMITH_API_KEY)
client.runs.wait(thread_id, ...)
```

The SDK converts that to an HTTP request that includes the API key:

```text
POST /threads/abc-123/runs/wait HTTP/1.1
Host: my-deployment.langgraph.app
x-api-key: lsv2_pt_...
Content-Type: application/json
```

## The authentication and authorization process.

Let's consider what authentication and authorization look like in a more general case with an authentication provider. There are three components: a client application, an auth provider, and your agent server.

1. **Client Application** — the frontend used by your users. It collects credentials, obtains a token from the auth provider, and includes that token in every request to your agent.

2. **Auth Provider (Identity Provider)** — a dedicated service that manages user identities, handles login, and issues signed tokens. Examples: Auth0, Supabase Auth.

3. **Agent Server** — your LangGraph application. It validates tokens through `@auth.authenticate`, enforces access control through `@auth.on.*`, and never stores credentials directly. The API server, mentioned previously, is a subcomponent of the Agent Server that runs and enforces the authentication and authorization.

An incoming request to your deployment from a logged-in user might look like this:

```text
POST /threads
Host: my-deployment.langgraph.app
x-api-key: ak_live_a8f3c2...
```

Every request to the Agent Server is authenticated. The notional diagram below shows the input path of a request.

<pre style="font-size:0.85rem;line-height:1.5;background:none;border:none;padding:0;">
HTTP request
     │
     ▼
  Uvicorn                        ← terminates HTTP, hands off to the Python app
     │
     ▼
 agent server
     │
     ├── auth                    ← runs on all LangGraph runtime requests *
     │   ├── @auth.authenticate
     │   └── @auth.on            ← runs per resource action (e.g. threads.create)
     │
     ├── custom middleware       ← optional
     │
     └── /runs, /threads, ...    ← agent server runtime
</pre>

> \* Custom routes bypass `@auth.authenticate` by default. Auth can be enabled on them via `"enable_custom_route_auth": true` in your `langgraph.json` HTTP config — but this applies to all custom routes. For selective auth (e.g., a public login route), declare a `path` parameter in your handler and branch on it.

You can replace the default `auth` step with your own handler — a Python function, decorated with `@auth.authenticate`, that runs on the agent server.

```python
from langgraph_sdk import Auth

auth = Auth()

@auth.authenticate
async def get_current_user(headers: dict) -> Auth.types.MinimalUserDict:
    # Validate credentials (e.g., API key, JWT token)
    api_key = headers.get(b"x-api-key")
    if not api_key or not is_valid_key(api_key):
        raise Auth.exceptions.HTTPException(
            status_code=401,
            detail="Invalid API key"
        )

    # Return user info — only identity is required.
    # Add any fields you want available downstream for authorization.
    return {
        "identity": "user-123",        # required — unique user identifier
        "email": "alice@example.com",  # optional — used in authorization
    }
```

The handler should:
1) Validate the credentials
2) On success, return a `MinimalUserDict` with at least the user's `identity`; on failure, raise   `Auth.exceptions.HTTPException` (e.g. 401)

This example reads the credential from the `headers` parameter. You can ask the decorator for other parameters too:

<details>
<summary>Decorator details — picking the parameters you want [click to expand]</summary>
<br>

The `@auth.authenticate` decorator inspects your function signature and injects whichever named parameters you ask for. The supported names are:

- `request` (`Request`) — the raw ASGI request object
- `path` (`str`) — the request path
- `method` (`str`) — the HTTP method (`GET`, `POST`, …)
- `path_params` (`dict[str, str]`) — URL path parameters
- `query_params` (`dict[str, str]`) — URL query parameters
- `headers` (`dict[bytes, bytes]`) — all request headers
- `authorization` (`str | None`) — convenience: the value of the `Authorization` header

You can mix and match. Later examples in this course use `authorization` because the credential always arrives in that header — but `headers: dict` is just as valid if you need access to other headers (e.g. a custom `x-api-key`). The name `authorization` is misleading — the header dates to HTTP/1.0 (1996) and actually carries authentication credentials, not authorization data.

See [Supported Parameters](https://docs.langchain.com/langsmith/auth#supported-parameters) in the LangSmith Auth docs for the full list.

</details>


LangGraph makes the authenticated user available in three places depending on where you are in the code:

| Where | How to read the user | Notes |
|---|---|---|
| An `@auth.on` handler | `ctx.user.identity` | Used in Custom Authorization (below). |
| A graph node receiving `config` | `config["configurable"]["langgraph_auth_user"]` | Standard pattern. |
| A graph node / middleware receiving `runtime` | `runtime.server_info.user.identity` | Same data, attribute access. `server_info` is `None` when running standalone. |


For example, inside a node:

```python
def my_node(state, config):
    user = config["configurable"].get("langgraph_auth_user")
    user["identity"]   # "user-123"
    user["email"]      # "alice@example.com"
```

Register the handler in `langgraph.json` so the deployment runs it:

```json
{
  "auth": {
    "path": "auth.py:auth"
  }
}
```

This is useful when your graph needs to personalize responses, enforce role-based logic, or call external APIs on the user's behalf. The next lesson walks through a complete working example.

Once the user's identity is established by `@auth.authenticate`, you can control what resources that user is allowed to access. This is done with `@auth.on` handlers, which are called after the `@auth.authenticate` handlers. The idea behind how this is done is pretty simple. Stamp resources with selected metadata when it's written. Use that same selected metadata as a filter on reads.

<div class="sd-wrap" id="ss-authz"></div>



An `@auth.on` handler runs after authentication and receives two arguments: the auth context (which includes the verified user) and the resource being accessed. It returns a filter that scopes the data to that user:

```python
# auth.py
@auth.on
async def add_owner(ctx: Auth.types.AuthContext, value: dict) -> dict:
    filters = {"owner": ctx.user.identity}
    metadata = value.setdefault("metadata", {})
    metadata.update(filters)  # updates value["metadata"]
    return filters
```

The returned filter is applied automatically — a user can only see threads, runs, and other resources that match it. This is what gives each user their own isolated view of data in a multi-tenant system.

You can also write handlers for specific resource types if different rules apply to different actions:

```python
# python/m5/l3b/auth.py
@auth.on.threads.create
async def on_thread_create(ctx, value):
    metadata = value.setdefault("metadata", {})  # get existing metadata dict, or create it if absent
    metadata["owner"] = ctx.user.identity        # stamp the new record in the database with the owner
    return {"owner": ctx.user.identity}    # filter: this user can only query records where owner matches

@auth.on.threads.read
async def on_thread_read(ctx, value):
    return {"owner": ctx.user.identity}    # filter: only return threads owned by this user
```

The most specific handler wins — so `@auth.on.threads.create` takes precedence over `@auth.on` for thread creation, while the general `@auth.on` handler covers everything else.

## What you learned in this lesson

- **AuthN vs AuthZ** — Authentication verifies who the user is; authorization determines what they can do.
- **`@auth.authenticate`** — a server-side Python handler that validates incoming tokens and returns a user dict with at least an `identity` field. LangGraph makes the full dict available to your graph nodes via `config["configurable"]["langgraph_auth_user"]`.
- **`@auth.on` handlers** — run after authentication and return a filter dict that LangGraph applies generically against each resource's `metadata` column. The key name is your convention — you stamp it on create and filter by it on read.
- **Handler specificity** — `@auth.on.threads.create` takes precedence over `@auth.on` for that specific action. The most specific handler wins.
- **Custom routes** — `@auth.authenticate` does not run on custom routes by default. It can be enabled globally via `"enable_custom_route_auth": true`, or handled selectively inside the handler itself using the `path` parameter.
