## JWT background

The token Supabase issues is a **JSON Web Token (JWT)** — three base64-url segments separated by dots: a header, a payload, and a signature. The payload carries claims about the user (identity, email, expiry) and the signature lets a verifier confirm the token was issued by Supabase and has not been tampered with.

For this lesson the important thing is that **`@auth.authenticate` does not verify the signature locally**. It calls Supabase's `/auth/v1/user` endpoint on every request to validate the token. This costs a network round-trip but gives you **revocation**: disabling a user or revoking a session in Supabase takes effect on the very next request to your deployment.

The alternative is local signature verification using Supabase's JWKS. It avoids the round-trip but cannot detect revoked tokens until they expire on their own. m5.4 and this lesson use the round-trip approach for clarity.

## How the tutor worked without auth

Before this lesson, `deep_tutor` had a stub authentication flow:

- **Dummy login** — `/login` looked up profiles by email alone. There was no password verification; anyone who knew an email address could "log in" as that user.
- **Open Resources tab** — `/resources` called `client.assistants.search()` and `client.threads.search()` without any filter. Every student's assistants and threads were returned, regardless of who was asking.
- **No Store isolation** — `client.store.get_item(("jane@example_com",), key="profile")` would return Jane's profile to anyone. The namespace was "private" only in that you had to know the email to construct it.

This was fine for a single-user development tutor but would not work in a real multi-tenant deployment.

## How auth changes this

| Before (deep_tutor) | After (m5/l5) |
|---|---|
| `/login` only checked the email | `/login` calls Supabase to validate password and issues a JWT |
| `@auth.authenticate` not registered | `@auth.authenticate` validates the JWT with Supabase on every request |
| `client.threads.search()` returned all threads | `@auth.on` filter scopes results to `metadata.owner == ctx.user.identity` |
| `client.assistants.search()` returned all assistants | Same — `@auth.on` filters by owner |
| Any client could read any namespace from the Store | `@auth.on.store()` asserts the caller can only access their own namespace |

The same UI runs in both modes — what changes is what the deployment lets the UI see.

## The auth handler

The handler is a small extension of m5.4: same JWT validation, same per-user ownership stamping, plus a store check that ensures users can only touch their own namespace.

<Tabs>
  <Tab title="Python">
```python {12-14,38,54-59}
# python/m5/l5/auth.py
import os
import httpx
from langgraph_sdk import Auth

auth = Auth()

SUPABASE_URL = os.environ["SUPABASE_URL"]
SUPABASE_SERVICE_KEY = os.environ["SUPABASE_SERVICE_KEY"]


def email_to_namespace(email: str) -> str:
    """Same conversion used by the UI's email_to_namespace."""
    return email.replace(".", "_")


@auth.authenticate
async def get_current_user(authorization: str | None) -> Auth.types.MinimalUserDict:
    """Validate JWT against Supabase and return the authenticated user."""
    if not authorization:
        raise Auth.exceptions.HTTPException(status_code=401, detail="Missing token")
    scheme, _, token = authorization.partition(" ")
    if scheme.lower() != "bearer":
        raise Auth.exceptions.HTTPException(status_code=401, detail="Invalid scheme")
    try:
        async with httpx.AsyncClient() as client:
            response = await client.get(
                f"{SUPABASE_URL}/auth/v1/user",
                headers={
                    "Authorization": authorization,
                    "apiKey": SUPABASE_SERVICE_KEY,
                },
            )
            response.raise_for_status()
            user = response.json()
            return {
                "identity": user["id"],
                "email": user["email"],
                "is_authenticated": True,
            }
    except Exception as e:
        raise Auth.exceptions.HTTPException(status_code=401, detail=str(e))


@auth.on
async def add_owner(ctx: Auth.types.AuthContext, value: dict):
    """Stamp owner on writes; return same filter so reads scope to this user."""
    filters = {"owner": ctx.user.identity}
    metadata = value.setdefault("metadata", {})
    metadata.update(filters)  # written to metadata on write/create actions
    return filters


@auth.on.store()
async def on_store(ctx: Auth.types.AuthContext, value: dict):
    """Only allow access to the namespace matching the caller's email."""
    expected = email_to_namespace(ctx.user["email"])
    namespace = value["namespace"]
    assert namespace[0] == expected, "Not authorized"
```
  </Tab>
  <Tab title="TypeScript">
```typescript {13-15,23-25,50,59-66}
// typescript/m5/l5/auth.ts
import { Auth, HTTPException } from "@langchain/langgraph-sdk/auth";

const SUPABASE_URL = process.env.SUPABASE_URL!;
const SUPABASE_SERVICE_KEY = process.env.SUPABASE_SERVICE_KEY!;

function emailToNamespace(email: string): string {
  return email.replace(/\./g, "_");
}

export const auth = new Auth()
  .authenticate(async (request) => {
    // TS langgraph-api applies .authenticate to ALL routes (including custom
    // /register and /login). Missing token → anonymous user so those routes can
    // run; the .on() handlers below still scope every data access by identity/email.
    const authorization = request.headers.get("Authorization");
    if (!authorization) {
      return { identity: "", email: "", is_authenticated: false, permissions: [] };
    }
    const [scheme] = authorization.split(" ");
    if (scheme?.toLowerCase() !== "bearer") {
      throw new HTTPException(401, { message: "Invalid scheme" });
    }
    try {
      const response = await fetch(`${SUPABASE_URL}/auth/v1/user`, {
        headers: { Authorization: authorization, apiKey: SUPABASE_SERVICE_KEY },
      });
      if (!response.ok) {
        throw new Error(`Supabase returned ${response.status}`);
      }
      const user = await response.json() as { id: string; email: string };
      return {
        identity: user.id,
        email: user.email,
        is_authenticated: true,
        permissions: [],
      };
    } catch (e) {
      throw new HTTPException(401, { message: String(e) });
    }
  })
  // Stamp owner on writes; return same filter so reads scope to this user.
  .on(["threads", "assistants"], ({ value, user }) => {
    const filters = { owner: user.identity };
    const v = value as { metadata?: Record<string, unknown> | null };
    v.metadata ??= {};
    (v.metadata as Record<string, unknown>).owner = user.identity;
    return filters;
  })
  // Only allow access to the namespace matching the caller's email.
  .on("store", ({ value, user }) => {
    const email = (user as unknown as { email?: string }).email ?? "";
    const expected = emailToNamespace(email);
    const namespace = value.namespace;
    if (!namespace || namespace[0] !== expected) {
      throw new HTTPException(403, { message: "Not authorized" });
    }
  });
```
  </Tab>
</Tabs>

The store check is the **bridge between auth identity and the email-derived namespace** the tutor uses everywhere else. The authenticate handler returns Supabase's UUID as `identity` — that's the standard auth identity. But the tutor indexes the Store by email (`jane@example_com` for Jane), and we want to keep that. So the store handler converts the caller's email back to the namespace form and asserts the caller is only touching their own.

<Tip>
**TypeScript: anonymous user on missing token**

In TypeScript, `langgraph-api` applies `.authenticate` to every route — including custom Hono routes like `/register` and `/login`. A new visitor has no token yet, so we return an anonymous user (`identity: ""`) instead of throwing 401. The `.on()` handlers still scope all data access; an anonymous user matches no owner filter, so they cannot read anyone else's data.
</Tip>

## The agent client

Until now, the client factory only accepted the LangSmith API key. With custom auth in place, every SDK call needs to carry a Supabase JWT instead. We make the JWT optional so the function still works in the old way when called without one:

<Tabs>

```python
# python/m5/l5/ui/agent_client.py (excerpt)
def create_client(deployment_url: str, jwt: str | None = None) -> LangGraphClient:
    if jwt:
        return get_client(url=deployment_url, headers={"Authorization": f"Bearer {jwt}"})
    api_key = os.environ.get("LANGSMITH_API_KEY", "")
    return get_client(url=deployment_url, api_key=api_key)
```

  </Tab>
  <Tab title="TypeScript">
```typescript
// typescript/m5/l5/ui/agentClient.ts (excerpt)
export function createClient(deploymentUrl: string, jwt?: string): Client {
  if (jwt) {
    return new Client({
      apiUrl: deploymentUrl,
      defaultHeaders: { Authorization: `Bearer ${jwt}` },
    });
  }
  const apiKey = process.env.LANGSMITH_API_KEY ?? "";
  return new Client({ apiUrl: deploymentUrl, apiKey });
}
```

```python
async def register(request: Request, req: RegisterRequest):
    # 1. Sign up with Supabase, receive a JWT.
    async with httpx.AsyncClient() as c:
        r = await c.post(
            f"{SUPABASE_URL}/auth/v1/signup",
```

```typescript
app.post("/register", async (c) => {
  const { first_name, last_name, email, password, goals = "" } = await c.req.json<{...}>();

  // 1. Sign up with Supabase, receive a JWT.
  const r = await fetch(`${SUPABASE_URL}/auth/v1/signup`, {

  const sessions = await createStudentSessions(client, namespace, `${first_name} ${last_name}`);
```

## `ui/index.html`

Two HTML changes: password fields on the login and register forms. Two JS changes: store the returned JWT on the browser, and send it as `Authorization: Bearer <jwt>` on `/chat` and `/resources` calls. The file is identical in both tracks.

```javascript
// m5/l5/ui/index.html (excerpt)
let state = {
  studentName: null,
  email: null,
  jwt: null,
  started: false,
  sessions: [],
};

function authHeaders() {
  return state.jwt ? { 'Authorization': `Bearer ${state.jwt}` } : {};
}

// /login response includes `jwt`; we save it and use it for every later call:
const data = await res.json();
state.jwt = data.jwt;
state.sessions = data.sessions;
saveState();

// Subsequent calls include the header:
await fetch('/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', ...authHeaders() },
  body: JSON.stringify({ ... }),
});
```

State is persisted to `localStorage`, so the JWT survives a page refresh until the user explicitly logs out.

### Try it

1. Click **register**. Enter a real-looking email + password + name + goals. Submit.
2. You should land on the chat UI with your name in the corner. Try a message in module-1 — the agent should respond.
3. Click **log out**, then **log in** with the same email + password. Your profile and conversation history should still be there — proof that the Store persisted them under your namespace.

   - The second user has their own empty conversation history.

### What's happening under the hood

Every backend call goes through this flow:

1. Browser sends the request with `Authorization: Bearer <supabase JWT>`.
2. The LangGraph agent server fires the authenticate handler first. It calls `GET https://<your-project>.supabase.co/auth/v1/user` with the bearer token and service-role key. Supabase returns the user's UUID and email if the token is valid; the handler returns `{ identity: uuid, email: email }`.
3. For thread/assistant operations, the `.on` ownership handler fires next. It stamps `metadata.owner = <uuid>` on writes and returns `{ owner: <uuid> }` as a filter on reads.
4. For Store operations, the store handler fires. It computes the expected namespace from the user's email and asserts the incoming namespace tuple starts with that value.
5. The agent runs under the authenticated user. Responses flow back the same way.

## What you learned

- **Wiring auth into a real application** — five files change: `auth.py`, `langgraph.json`, `ui/agent_client.py`, `ui/app.py`, `ui/index.html`. Each has a focused responsibility.
- **Decoupling auth identity from data identity** — Supabase issues UUIDs; the tutor's Store is indexed by email-derived namespaces. `@auth.on.store()` bridges them at the only place they need to meet.
- **Browser-side JWT handling** — store the token after login, attach it as a Bearer header on every subsequent request, clear it on logout.
- **Revocation via the round-trip pattern** — calling `/auth/v1/user` on every request costs latency but means revoked tokens stop working immediately.
