## JWT background

The token Supabase issues is a **JSON Web Token (JWT)** ā€” three base64-url segments separated by dots: a header, a payload, and a signature. The payload carries claims about the user (identity, email, expiry) and the signature lets a verifier confirm the token was issued by Supabase and has not been tampered with.

For this lesson the important thing is that **`@auth.authenticate` does not verify the signature locally**. It calls Supabase's `/auth/v1/user` endpoint on every request to validate the token. This costs a network round-trip but gives you **revocation**: disabling a user or revoking a session in Supabase takes effect on the very next request to your deployment.

The alternative is local signature verification using Supabase's JWKS. It avoids the round-trip but cannot detect revoked tokens until they expire on their own. m5.4 and this lesson use the round-trip approach for clarity.

## How the tutor worked without auth

Before this lesson, `deep_tutor` had a stub authentication flow:

- **Dummy login** ā€” `/login` looked up profiles by email alone. There was no password verification; anyone who knew an email address could "log in" as that user.
- **Open Resources tab** ā€” `/resources` called `client.assistants.search()` and `client.threads.search()` without any filter. Every student's assistants and threads were returned, regardless of who was asking.
- **No Store isolation** ā€” `client.store.get_item(("jane@example_com",), key="profile")` would return Jane's profile to anyone. The namespace was "private" only in that you had to know the email to construct it.

This was fine for a single-user development tutor but would not work in a real multi-tenant deployment.

## How auth changes this

| Before (deep_tutor) | After (m5/l5) |
|---|---|
| `/login` only checked the email | `/login` calls Supabase to validate password and issues a JWT |
| `@auth.authenticate` not registered | `@auth.authenticate` validates the JWT with Supabase on every request |
| `client.threads.search()` returned all threads | `@auth.on` filter scopes results to `metadata.owner == ctx.user.identity` |
| `client.assistants.search()` returned all assistants | Same ā€” `@auth.on` filters by owner |
| Any client could read any namespace from the Store | `@auth.on.store()` asserts the caller can only access their own namespace |

The same UI runs in both modes ā€” what changes is what the deployment lets the UI see.

## `auth.py`

The handler is a small extension of m5.4: same JWT validation, same `@auth.on` ownership stamping, plus an `@auth.on.store()` check that ensures users can only touch their own namespace.

```python {12-14,38,54-59}
# python/m5/l5/auth.py
import os
import httpx
from langgraph_sdk import Auth

auth = Auth()

SUPABASE_URL = os.environ["SUPABASE_URL"]
SUPABASE_SERVICE_KEY = os.environ["SUPABASE_SERVICE_KEY"]


def email_to_namespace(email: str) -> str:
    """Same conversion used by the UI's email_to_namespace."""
    return email.replace(".", "_")


@auth.authenticate
async def get_current_user(authorization: str | None) -> Auth.types.MinimalUserDict:
    """Validate JWT against Supabase and return the authenticated user."""
    if not authorization:
        raise Auth.exceptions.HTTPException(status_code=401, detail="Missing token")
    scheme, _, token = authorization.partition(" ")
    if scheme.lower() != "bearer":
        raise Auth.exceptions.HTTPException(status_code=401, detail="Invalid scheme")
    try:
        async with httpx.AsyncClient() as client:
            response = await client.get(
                f"{SUPABASE_URL}/auth/v1/user",
                headers={
                    "Authorization": authorization,
                    "apiKey": SUPABASE_SERVICE_KEY,
                },
            )
            response.raise_for_status()
            user = response.json()
            return {
                "identity": user["id"],
                "email": user["email"],
                "is_authenticated": True,
            }
    except Exception as e:
        raise Auth.exceptions.HTTPException(status_code=401, detail=str(e))


@auth.on
async def add_owner(ctx: Auth.types.AuthContext, value: dict):
    """Stamp owner on writes; return same filter so reads scope to this user."""
    filters = {"owner": ctx.user.identity}
    metadata = value.setdefault("metadata", {})
    metadata.update(filters)  # written to metadata on write/create actions
    return filters


@auth.on.store()
async def on_store(ctx: Auth.types.AuthContext, value: dict):
    """Only allow access to the namespace matching the caller's email."""
    expected = email_to_namespace(ctx.user["email"])
    namespace = value["namespace"]
    assert namespace[0] == expected, "Not authorized"
```

The store check is the **bridge between auth identity and the email-derived namespace** the tutor uses everywhere else. `@auth.authenticate` returns Supabase's UUID as `identity` ā€” that's the standard auth identity. But `deep_tutor` indexes the Store by email (`jane@example_com` for Jane), and we want to keep that. So `@auth.on.store()` converts the caller's email back to the namespace form and asserts the caller is only touching their own.

## `ui/agent_client.py`

Until now, `create_client` only accepted the LangSmith API key. With custom auth in place, every SDK call needs to carry a Supabase JWT instead. We make the JWT optional so the function still works in the old way when called without one:

```python
# python/m5/l5/ui/agent_client.py (excerpt)
def create_client(deployment_url: str, jwt: str | None = None) -> LangGraphClient:
    if jwt:
        return get_client(url=deployment_url, headers={"Authorization": f"Bearer {jwt}"})
    api_key = os.environ.get("LANGSMITH_API_KEY", "")
    return get_client(url=deployment_url, api_key=api_key)
```

When a JWT is passed, every SDK call sends it as a Bearer token in the `Authorization` header ā€” exactly what `@auth.authenticate` expects.

## `ui/app.py`

The biggest file. `/register` and `/login` now actually talk to Supabase:

```python
# python/m5/l5/ui/app.py (register excerpt)
@app.post("/register")
async def register(request: Request, req: RegisterRequest):
    # 1. Sign up with Supabase, receive a JWT.
    async with httpx.AsyncClient() as c:
        r = await c.post(
            f"{SUPABASE_URL}/auth/v1/signup",
            json={"email": req.email, "password": req.password},
            headers={"apiKey": SUPABASE_ANON_KEY, "Content-Type": "application/json"},
        )
        if not r.is_success:
            raise HTTPException(status_code=r.status_code, detail=r.json())
        jwt = r.json()["access_token"]

    # 2. Use that JWT for every subsequent SDK call.
    client = create_client(get_deployment_url(request), jwt=jwt)
    namespace = email_to_namespace(req.email)

    await client.store.put_item((namespace,), key="profile", value={...})
    sessions = await create_student_sessions(client, ..., namespace=namespace)
    await client.store.put_item((namespace,), key="sessions", value={"sessions": sessions})

    return {"jwt": jwt, "sessions": sessions, "profile": {...}}
```

`/login` follows the same pattern, hitting Supabase's `/auth/v1/token?grant_type=password` endpoint and then reading the profile + sessions back from the Store with the returned JWT.

`/chat` and `/resources` extract the JWT from the incoming `Authorization` header and pass it to the SDK client:

```python
# python/m5/l5/ui/app.py (chat excerpt)
@app.post("/chat")
async def chat(
    request: Request,
    req: ChatRequest,
    authorization: str | None = Header(default=None),
):
    jwt = extract_jwt(authorization)
    client = create_client(get_deployment_url(request), jwt=jwt)
    return StreamingResponse(
        stream_response(client, req.thread_id, req.assistant_id, req.message),
        media_type="text/plain",
    )
```

Every call to LangGraph now carries the user's JWT. `@auth.authenticate` validates it with Supabase, `@auth.on` filters resources by `owner = ctx.user.identity`, and `@auth.on.store()` enforces the namespace match.

## `ui/index.html`

Two HTML changes: password fields on the login and register forms. Two JS changes: store the returned JWT on the browser, and send it as `Authorization: Bearer <jwt>` on `/chat` and `/resources` calls.

```javascript
// python/m5/l5/ui/index.html (excerpt)
let state = {
  studentName: null,
  email: null,
  jwt: null,
  started: false,
  sessions: [],
};

function authHeaders() {
  return state.jwt ? { 'Authorization': `Bearer ${state.jwt}` } : {};
}

// /login response includes `jwt`; we save it and use it for every later call:
const data = await res.json();
state.jwt = data.jwt;
state.sessions = data.sessions;
saveState();

// Subsequent calls include the header:
await fetch('/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', ...authHeaders() },
  body: JSON.stringify({ ... }),
});
```

State is persisted to `localStorage`, so the JWT survives a page refresh until the user explicitly logs out.

### What's happening under the hood

Every backend call goes through this flow:

1. Browser sends the request with `Authorization: Bearer <supabase JWT>`.
2. The LangGraph agent server fires `@auth.authenticate` first. It calls `GET https://<your-project>.supabase.co/auth/v1/user` with the bearer token and service-role key. Supabase returns the user's UUID and email if the token is valid; the handler returns `{"identity": uuid, "email": email}`.
3. For thread/assistant operations, `@auth.on` fires next. It stamps `metadata.owner = <uuid>` on writes and returns `{"owner": <uuid>}` as a filter on reads.
4. For Store operations, `@auth.on.store()` fires. It computes the expected namespace from `ctx.user["email"]` and asserts the incoming namespace tuple starts with that value.
5. The agent runs under the authenticated user. Responses flow back the same way.

## What you learned

- **Wiring auth into a real application** ā€” five files change: `auth.py`, `langgraph.json`, `ui/agent_client.py`, `ui/app.py`, `ui/index.html`. Each has a focused responsibility.
- **Decoupling auth identity from data identity** ā€” Supabase issues UUIDs; the tutor's Store is indexed by email-derived namespaces. `@auth.on.store()` bridges them at the only place they need to meet.
- **Browser-side JWT handling** ā€” store the token after login, attach it as a Bearer header on every subsequent request, clear it on logout.
- **Revocation via the round-trip pattern** ā€” calling `/auth/v1/user` on every request costs latency but means revoked tokens stop working immediately.
